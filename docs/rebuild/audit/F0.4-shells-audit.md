## 1. Diagrama del flujo actual

```
ReactDOM.createRoot(...)                              [main.jsx]
  └── <AppErrorBoundary>
       └── <DeviceProfileProvider>                    [hooks: useResponsiveLayout → theme, viewport, shellFamily, inputMode, shellPolicy]
            └── <App>                                  [hooks de dominio: useAssessmentFlow, useAuthSession, useChatSession, useCourseProgress, useRemoteProgressSync]
                 ├── useDeviceProfile() ← context
                 ├── const routeDefinition = resolveActiveRoute({...})   → { routeKey, routeMeta, slots: { primary, secondary } }
                 │     ├── auth.sessionLoading?  → SessionLoadingRouteContainer
                 │     ├── !auth.currentUser?    → PublicRouteContainer  → AuthView
                 │     └── autenticado?          → AuthenticatedRouteContainer  → { SurveyView | CoursesView | LessonView | AdminView }
                 ├── const navigation = buildNavigationModel({...})
                 ├── const shellSlots = buildShellSlots({...})           → { header (SessionBar), primary, secondary, floating (chat FAB), overlay (ChatDrawer) }
                 └── <DeviceShell shellFamily, routeKey, routeIntent, header, primary, secondary, floating, overlay />
                      └── renderShell = SHELL_RENDERERS[shellFamily]     → MobileShell | TabletShell | DesktopShell  (llamada como función, no como JSX)
                           └── retorna árbol macro con: data-shell-family, data-shell-intent, data-shell-*-mode, data-shell-slot="header|primary|secondary|floating|overlay"
```

Puntos clave del flujo:
- Hay **una sola base lógica compartida** en `App` — correcto según la tesis de 05-shell-contract.md ("hay una sola base lógica compartida").
- `DeviceProfileProvider` traduce viewport → `shellFamily` + `shellPolicy` y los expone por context. Correcto según el nombre conceptual `DeviceProfileProvider` del contrato.
- `resolveActiveRoute` ejerce el rol conceptual `RouteContainer` del contrato — decide qué vista de dominio mostrar.
- `DeviceShell` ejerce el rol conceptual `DeviceShell` del contrato — aplica organización macro.
- Las vistas (`AuthView`, `SurveyView`, `CoursesView`, `LessonView`, `AdminView`) son `PresentationalSections`.

El diagrama conceptual del contrato (§Piezas conceptuales) está implementado **con los cinco roles distinguidos** (AppCoordinator ≡ App, DeviceProfileProvider, RouteContainer, DeviceShell, PresentationalSections).

**Observación menor:** en `DeviceShell.jsx:26` se usa `return renderShell({...})` en vez de `<MobileShell {...} />`. Esto invoca el componente como función y fusiona sus hooks con los de DeviceShell. Funciona porque ningún shell usa hooks — si alguien añadiera `useEffect` dentro de `MobileShell`, se comportaría como hook de `DeviceShell` y se reordenaría al cambiar de familia, causando el error "hooks must be called in the same order". **Code smell con bomba futura.** El patrón correcto es `const Shell = SHELL_RENDERERS[...]; return <Shell ... />;`.

## 2. Preservación de estado al cambiar viewport

### 2.1 Guarantías del contrato
El contrato (05-shell-contract.md §Garantías no negociables) exige que cambiar de shell **no reset** de: auth, assessment, ruta, progreso, `currentLesson`, chat abierto/cerrado, admin preview.

### 2.2 Evidencia por código
Todo el estado que enumera el contrato vive en **hooks de `App`**, fuera de los shells:

| Estado | Dónde vive | Sobrevive al cambio de shell? |
|---|---|---|
| Autenticación | `useAuthSession` en App:60 | ✅ App no remonta al cambiar viewport |
| Assessment | `useAssessmentFlow` en App:35 | ✅ |
| Plan de curso y progreso | `useCourseProgress` en App:23 | ✅ |
| `currentView` / `currentLesson` | Parte de `useCourseProgress` | ✅ |
| Chat (mensajes, input, abierto) | `useChatSession` en App:22 | ✅ |
| Sync remoto | `useRemoteProgressSync` en App:102 | ✅ |
| Admin preview toggle | `useState` en App:33 | ✅ |
| Analytics de admin | `useState` en App:30-32 | ✅ |
| Theme | `useState` en `useResponsiveLayout:64` | ✅ persiste en localStorage |

### 2.3 Lo que SÍ resetea al cambiar shell
`DeviceShell.jsx:26` invoca `renderShell(...)` que retorna la JSX de Mobile/Tablet/Desktop. Los tres shells retornan estructuras JSX **diferentes**:

- MobileShell: `<div class="app-shell-mobile">…<div class="grid min-h-screen grid-rows-[auto_1fr]">…<main data-shell-slot="primary">{primary}</main>…</div></div>`
- TabletShell: `<div class="app-shell-tablet">…<div class="grid gap-..."><div class="sd-page-shell grid items-start gap-..."><main>{primary}</main>{secondary && <aside>…</aside>}</div>…</div>`
- DesktopShell: `<div class="app-shell-desktop">…<div class="sd-page-shell" data-shell-slot="body">{secondary ? <div class="grid items-start gap-..."><main>{primary}</main><aside>{secondary}</aside></div> : <main>{primary}</main>}</div>`

Al cambiar `shellFamily`, la profundidad del DOM en la que vive `<main>{primary}</main>` cambia (tablet y desktop lo anidan un nivel más dentro del grid). El reconciler de React trata esto como **unmount + remount del árbol de `primary`**. Consecuencias:

- Los componentes de vista (`SurveyView`, `CoursesView`, `LessonView`) **remontan**.
- Todo `useState` / `useRef` interno de esas vistas se reinicia.
- El scroll position se pierde (además hay un `window.scrollTo({ top: 0 })` en App:162 al cambiar `currentView` — independiente de esto).
- Si había animaciones en curso (p.ej. chat typing, progress bar animando), se cortan.

**Veredicto contrato vs. realidad:**
- Los 6 requisitos duros del contrato (auth, assessment, ruta, progreso, currentLesson, chat open, admin preview) **se cumplen** porque todo está elevado a App.
- Estado local de vistas **NO se preserva**. Si `SurveyView` guarda internamente (p.ej.) el índice de scroll, la validación visible o un toggle de "mostrar explicación", se pierde al rotar.
- Simulaciones: si el usuario está en medio de `signalActivities` (6A, chat) y rota el dispositivo de tablet a laptop, la simulación **remonta**. Dado que el contrato 06 exige preservar scoring/feedback/progress, y esos datos viven en `useCourseProgress` (App), el scoring sobrevive. Pero la UI de la conversación en curso (mensajes mostrados, animación typing) se reinicia visualmente.

Esto técnicamente **cumple la letra del contrato**, pero no el espíritu completo. No es bloqueo, es banderazo para F2 si se optimiza.

## 3. Cumplimiento de slots del contrato

El contrato 05 no enumera una lista cerrada de slots obligatorios, pero las piezas típicas de un shell son: Header, Nav, Main, Aside, Footer. El playbook de F2 (03-phase-prompt-templates.md) habla de "shell contract implementado". Los slots actuales son:

| Slot implementado | Attr `data-shell-slot` | Mobile | Tablet | Desktop |
|---|---|---|---|---|
| `header` | `"header"` | ✅ sticky top | ✅ dentro de `sd-page-shell` | ✅ dentro de `sd-page-shell` |
| `primary` | `"primary"` (elemento `<main>`) | ✅ single-task | ✅ dentro de grid | ✅ dentro de grid |
| `secondary` | `"secondary"` (elemento `<aside>` en Tablet/Desktop, `<div>` en Mobile) | ✅ stack vertical (si presente) | ✅ split condicional (`lg:grid-cols-[...]` si `routeIntent !== 'focus'`) | ✅ aside sticky (`xl:grid-cols-[...]` si hay secondary) |
| `floating` | `"floating"` | ✅ bottom-right fixed | ✅ bottom-right fixed | ✅ bottom-right fixed |
| `overlay` | `"overlay"` | ✅ fullscreen capa | ✅ | ✅ |

**Slots mencionados en el contrato estándar y no implementados como slot propio:**
- **Nav** — no existe un slot separado para navegación. Los nav items viven **dentro** del `header` (SessionBar los renderiza). Para un producto con nav lateral persistente en desktop (patrón típico de dashboards), esto obligaría a reestructurar.
- **Footer** — no existe. Ninguna ruta actualmente requiere footer, pero cualquier vista que lo necesite (legal, educativo) tendría que improvisarlo.
- **Body contextual / aside-left** — no existe. En desktop, el patrón "rail izquierdo de navegación + main + aside derecho" no es expresable; solo hay `primary + secondary (aside derecho)`.

### 3.1 `<main>` aparece una sola vez
Cada shell envuelve `primary` en `<main data-shell-slot="primary">`. Esto es correcto WAI-ARIA (debe haber un solo `<main>`). ✅

### 3.2 `<aside>` solo en Tablet/Desktop
- MobileShell usa `<div>` para `secondary` (línea 34), no `<aside>`. Semántica correcta — en móvil no hay distinción aside vs. stacked content. ✅
- TabletShell (línea 40) y DesktopShell (línea 38) sí usan `<aside>`. ✅

### 3.3 Overlays y floating
- Overlay usa `style={{ zIndex: 'var(--sd-z-overlay)' }}` — correcto en los 3 shells.
- Floating usa `style={{ zIndex: 'var(--sd-z-drawer)' }}` — inconsistente con el nombre "floating" (es el botón de chat, no un drawer).
- En móvil el botón flotante está en `bottom-5 right-5`; en tablet/desktop en `bottom-6 right-6`. Diferencia de 4px, razonable.

### 3.4 Policies aplicadas como data-attrs
`shellPolicies.js` exporta 3 objetos con `headerMode`, `primaryMode`, `secondaryMode`, `secondaryPersistent`, `overlayMode`, `floatingMode`, `slotOrder`. Se aplican como `data-shell-*-mode` en el contenedor raíz de cada shell. **Útil para hojas de estilo y para test de snapshot**. `secondaryPersistent: true` solo en desktop.

La policy ES una fuente de verdad declarativa — buen diseño.

### 3.5 Detalle de atributos
Cada shell aplica:
- `data-shell-family` (mobile/tablet/desktop)
- `data-shell-intent` (focus/workspace/immersive — viene de `routeMeta.shellIntent`)
- `data-shell-primary-mode`, `data-shell-secondary-mode`, `data-shell-overlay-mode`
- clase `app-shell app-shell-{family} app-shell-family-{family}` (¡duplicado: `app-shell-mobile` y `app-shell-family-mobile` aplican lo mismo!)
- clase `app-view-{routeKey}`

**Duplicado redundante:** `app-shell-mobile` y `app-shell-family-mobile` — las dos clases coexisten en el className. Es un vestigio probable de la transición del sistema viejo al nuevo; `app.css` puede tener reglas para `app-shell-mobile` y el sistema nuevo para `app-shell-family-mobile`. A limpiar en F8.

### 4.2 ¿Lo consume algo fuera del legacy?

- **0 matches en código JS/JSX nuevo o CSS nuevo** (`tailwind.css`).

Grep `body[data-theme]`:
- **17 matches en `tailwind.css`** (dark mode styling por selector `body[data-theme='dark']`).

Grep `body[data-shell]`:

Grep `body[data-inputMode]` (con variaciones `input-mode`):
- Pocos matches; habría que inspeccionar app.css con detalle para confirmar.

### 4.3 Resumen
- `body.dataset.theme` — **necesario** para tailwind.css dark mode (17 selectores). Hasta que haya otro mecanismo (p.ej. `@media (prefers-color-scheme)` o clase en `html`), no se puede quitar.
- `body.dataset.viewport` — **solo lo usa `app.css` legacy** (64 selectores). Está vivo porque legacy CSS no se ha migrado. A retirar cuando se elimine `app.css` (F8).
- `body.dataset.shell` — **dead write**. Nadie lo lee. Se puede quitar ahora sin riesgo.
- `body.dataset.inputMode` — a verificar; probablemente también legacy-only.

**Banderín:** el contrato 05 identificó explícitamente `body.dataset.viewport` como riesgo pendiente ("Ambigüedad abierta"). Sigue vigente. El shell contract sigue acoplado a CSS legacy vía esta puerta trasera, y romper el acoplamiento requiere limpiar `app.css` primero.

## 5. Breakpoints actuales vs. recomendados

### 5.1 Breakpoints implementados
`useResponsiveLayout.js:16-24` — `getViewportProfile(width)`:

| Rango (inclusive) | Perfil fino | Shell family |
|---|---|---|
| 0–420 px | `phone-small` | mobile |
| 421–640 px | `phone` | mobile |
| 641–820 px | `tablet-compact` | tablet |
| 821–1024 px | `tablet` | tablet |
| 1025–1280 px | `laptop` | desktop |
| 1281+ px | `desktop` | desktop |

Los media queries en `tailwind.css` usan solo dos breakpoints propios:
- `@media (min-width: 48rem)` = 768 px → shell tablet takeover
- `@media (min-width: 80rem)` = 1280 px → shell desktop takeover

Es decir: el **CSS** cambia shell spacing a 768 y a 1280, mientras que el **JS** cambia shell family a 640 y a 1024. **Inconsistencia**.

### 5.2 Breakpoints recomendados por el prompt
`320–767` / `768–1023` / `1024+`

### 5.3 Comparación

| Breakpoint | Recomendado (JS) | Implementado (JS) | Implementado (CSS) | Drift |
|---|---|---|---|---|
| Mobile cutoff | 767 | 640 | 768 | JS 128 px **antes**, CSS 1 px después |
| Tablet cutoff | 1023 | 1024 | 1280 | JS coincide; CSS 257 px **después** |
| Desktop arranque | 1024 | 1025 | 1280 | JS coincide; CSS 256 px **después** |

**Zonas problemáticas:**
1. **640–767 px** — el JS ya dice "tablet" (usa TabletShell), pero el CSS todavía está en `:root` (padding mobile) hasta 767. **El usuario ve TabletShell con paddings de móvil**. Visualmente inconsistente en teléfonos grandes en landscape o tabletas en portrait pequeñas.
2. **1024–1280 px** — el JS ya dice "desktop" (usa DesktopShell), pero el CSS sigue en tablet padding hasta 1280. **DesktopShell con paddings de tablet**. Laptops en ventana reducida se ven apretados.

Esto explicaría parte del 🔶 "CSS con paddings incómodos" si F0.7 detecta regresiones de layout al testear en laptop (1100–1200 px).

### 5.4 Recomendación
Alinear los 3 bordes: 767/1023/1024 en JS **y** 768/1280→768/1024 en CSS. O al revés (mantener 640 / 1024 en JS y alinear CSS a 640/1024). Cualquier opción está bien — la actual es inconsistente.

## 7. Veredicto por shell

### 7.1 MobileShell — ✅
- Orden claro: header sticky + main single-task + secondary opcional stack + floating fixed + overlay fullscreen.
- Cumple contrato `MobileShell`: chrome mínimo, una tarea principal, sin panel lateral persistente.
- `sd-page-shell` solo se aplica al header (no al primary) — esto hace que el primary ocupe todo el ancho. Decisión intencional; si el contenido del primary usa `sd-page-shell` internamente (como hacen las vistas), el padding aparece ahí.
- Safe-area insets: no aplica bottom/top safe-area a `fixed bottom-5 right-5`. En notch phones con gesture bar, el FAB puede solaparse. Menor.
- Duplica clase `app-shell-mobile` y `app-shell-family-mobile` — transitorio.

### 7.2 TabletShell — ✅ con un matiz
- Split condicional (`lg:grid-cols-[...]`) solo cuando `routeIntent !== 'focus'` y hay `secondary`. Correcto según contrato.
- La condición `allowSplit = Boolean(secondary) && routeIntent !== 'focus'` está bien, pero `lg:` en Tailwind es 64rem = 1024 px, lo que en un tablet 820–1023 px (dentro del rango tablet del JS) **el split no se activa**, y el secondary se apila debajo. En la práctica split real solo aparece en tablet grande (iPad landscape, ~1024+). Efectivamente: **el split nunca ocurre en devices que el JS clasifica como tablet**, porque el rango tablet va hasta 1024 y el split se activa a 1024. Un pixel de diferencia — casi siempre apilado.
- En la vista apilada, el `<aside>` usa `grid gap-[var(--sd-shell-rail-gap)]` — correcto.

### 7.3 DesktopShell — ✅
- Aside sticky vía `sd-rail-sticky` (definido en tailwind.css:1153 usando `position: sticky; top: calc(var(--sd-shell-padding-block) + var(--sd-space-2))`). Bien.
- Grid solo se activa en `xl:` (80rem = 1280 px). Para 1024–1280 px (laptop en el bucket desktop del JS), el aside se apila también (el `xl:` no aplica). Igual que en tablet, hay una franja donde "shell desktop sin split real". Razonable pero amplifica el drift JS/CSS del §5.
- `focusedPrimary` limita el main a `max-w-[72rem]` centrado — útil para lectura en desktops ultrawide.

### 7.4 DeviceShell — 🔶
- Dispatcher correcto pero el patrón `return renderShell({...})` (línea 26) es una llamada de función, no un render de componente. Sin daño actual (los 3 shells son pure functions sin hooks), pero **deja una bomba si en el futuro se añaden hooks a los shells**. 🔶.

### 7.5 Navigation (`navigationPolicy.js` + `SessionBar`) — ✅
- Items base: `survey`, `courses` + `admin` si `isAdmin`. Sencillo.
- `normalizeRequestedView` mapea `lesson` → `courses` (porque lesson no es destino navegable directo). Correcto.
- `getRouteMeta` retorna metadatos de cada ruta (label, shellIntent, showHeader, allowChat, navigationTarget).
- `buildNavigationModel` retorna items con flag `active` basado en `navigationTarget`.
- Tests cubiertos en `test/shell-architecture.test.js` y `test/responsive-smoke.test.js`.

## 8. Micro-hallazgos y deuda identificada

1. **`DeviceShell.jsx:26` render-as-function** — code smell con potencial de error de hooks futuro.
2. **Doble wrap `sd-page-shell`** — TabletShell y DesktopShell ya envuelven body en `sd-page-shell`, y las vistas (`CoursesView:152`, `LessonView:92`, `SurveyView:923`) repiten la clase dentro. Funciona (max-width del anidado es igual o menor), pero es redundante.
3. **Clases duplicadas `app-shell-<family>` + `app-shell-family-<family>`** — vestigio de transición; limpiar en F8.
4. **`body.dataset.shell` es dead write** — nadie lo lee. Quitar del hook.
5. **Drift breakpoints JS ↔ CSS** — JS cambia shell a 640/1024, CSS a 768/1280. Crea dos franjas de 128 y 256 px donde la shell family dice X y el paddings dicen Y.
6. **`ProgressSummary` sin `data-sd-container`** — bug silencioso: la `@container (min-width: 54rem)` de progress summary nunca se activa.
7. **Z-index de `floating` usa `--sd-z-drawer`** (50) pero el FAB no es drawer. Debería ser un token propio `--sd-z-fab` o al menos `--sd-z-sticky` (20).
8. **Safe-area insets no contempladas** — en mobile con notch/gesture bar, el FAB (`bottom-5`) puede solapar el home bar del sistema.
9. **`SessionLoadingRouteContainer.jsx:14`** hardcodea el texto "Restaurando sesión" sin variante i18n ni prop para customizar. Menor.
10. **`AuthenticatedRouteContainer.jsx:32`** tiene fallback a `SurveyView` si `content` es nulo después del if-else — implica estado inválido aceptable. Bien para defensive programming.
11. **`resolveActiveRoute.jsx:63`** — `routeMeta = getRouteMeta(course.currentView, { isAdmin })` — si `currentView === 'lesson'` devuelve meta de `lesson`, pero `AuthenticatedRouteContainer` renderiza `LessonView` cuando `currentView === 'lesson'`. OK. El `normalizeRequestedView` solo se invoca en App para cambio de view; internamente el flujo usa el `currentView` directo.
12. **Tests de shell-architecture** (line 14) cubren: `buildResponsiveProfile`, `getInputMode`, `getShellFamily`, `normalizeRequestedView`, `getRouteMeta`, `buildNavigationModel`, `getShellLayoutPolicy`. **No testean** el render real de los shells ni la conservación de estado al cambiar shellFamily.

## 9. Riesgos para fases siguientes

1. **F2 técnicamente cerrable**, pero con 3 cabos pendientes:
   - Patrón `return renderShell(...)` en DeviceShell debe volverse `<Shell />`.
   - Alineación breakpoints JS ↔ CSS.
   - Container-awareness de `ProgressSummary`.
2. **F4 (Dashboard)**: `CoursesView` usa múltiples secondary panels y transiciones tabla/card. El shell no expone slot propio para "rail izquierdo" — si F4 necesita un side-nav, habrá que extender el contrato de shells.
3. **F5 (Lesson) + F6 (Simulaciones)**: el `routeIntent: 'immersive'` está definido en `navigationPolicy.js:38` para lesson, pero **nunca se interpreta** en los shells. El immersive necesita más que eso — debería desactivar rail, dar full-screen en mobile, etc. Lagunar.
4. **F7 (Chat + Admin)**: ChatDrawer hoy está como `overlay` del shell. Cuando F7 lo migre a new UI, el shell ya tiene el slot listo — bien. AdminView está dentro de `primary` sin secondary — si admin necesita un rail propio, habrá que mover su layout a secondary slot explícitamente.
