# F6.R10 - Specific simulation passes

Fecha: 2026-04-24
Estado: cerrada localmente

## Objetivo

Refinar las simulaciones por canal despues de los pases transversales de contraste, densidad, identidad y fullscreen.

F6.R10 no reabre route, Survey, Lesson shell, dominio, scoring ni contratos. El foco es que cada simulacion tenga una accion mas clara, una identidad mas propia y affordances visibles sin meter paneles grandes ni texto largo.

## Cambios implementados

### 1. WhatsApp / chat

Archivo: `frontend/src/components/activities/signalActivities.jsx`

Se agrego:

- `data-sd-specific-simulation-pass="chat"`
- `sd-chat-stage-cues`
- `data-sd-specific-strip="chat"`

La tira compacta muestra:

- senal dominante;
- estado no verificado;
- salida por fuera.

Resultado:

- el chat comunica mas rapido que no se debe resolver dentro del canal;
- el stage gana una clave visual sin abrir otro panel lateral;
- se conserva el composer y el hilo como foco principal.

### 2. SMS / email inbox

Archivo: `frontend/src/components/activities/immersive/InboxActivity.jsx`

Se agrego:

- `data-sd-specific-simulation-pass={simulationCategory}`;
- `inbox-action-strip`;
- `data-sd-specific-strip="inbox"`.

La tira compacta marca el flujo:

- abrir;
- clasificar;
- evaluar.

Resultado:

- SMS y correo quedan mas guiados sin meter texto largo;
- la accion principal se entiende antes de entrar al detalle;
- el rail/lista sigue subordinado al lector.

### 3. WebLab

Archivo: `frontend/src/components/activities/immersive/WebLabActivity.jsx`

Se agrego:

- `data-sd-specific-simulation-pass="weblab"`;
- `web-lab-stage-tabs`;
- `data-sd-specific-strip="weblab"`;
- `aria-current="step"` en la etapa activa.

Resultado:

- producto, carrito y checkout se leen como etapas reales del sitio;
- el navegador deja de sentirse como botones sueltos;
- el stage conserva la metafora web sin agregar copy.

### 4. ScenarioFlow

Archivo: `frontend/src/components/activities/immersive/ScenarioFlowActivity.jsx`

Se agrego:

- `data-sd-specific-simulation-pass="scenario"`;
- `scenario-decision-card`;
- `scenario-choice-stack`;
- `scenario-option-card`;
- `scenario-flow-rail`;
- `data-sd-specific-strip="scenario"`.

Resultado:

- la decision activa se vuelve el centro visual del flujo;
- las opciones se leen como tarjetas propias del escenario;
- la bitacora queda mas claramente subordinada al lado.

### 5. Calls

Archivo: `frontend/src/components/activities/CallSimulationActivity.jsx`

Se agrego:

- `data-sd-specific-simulation-pass="call"`;
- `data-sd-call-phase={phase}`;
- `call-safety-strip`;
- `data-sd-specific-strip="call"`;
- `data-sd-call-control-dock="true"` en el dock de controles.

Las tiras compactas refuerzan:

- limite claro;
- cortar presion;
- verificar fuera;
- no dar codigos;
- colgar si insiste.

Resultado:

- la llamada comunica mejor su estado y su regla de seguridad;
- los controles se sienten como un dock de llamada, no como botones sueltos;
- la urgencia del canal queda mas visible sin tocar la logica de voz/IA.

### 6. Concept / analysis

Archivo: `frontend/src/components/activities/signalActivities.jsx`

Se agrego:

- `analysis-action-strip`;
- `data-sd-specific-strip="analysis"` en CompareDomains y SignalHunt.

Resultado:

- las actividades analiticas tienen un ritmo visible propio;
- comparar, elegir, marcar y evaluar queda mas claro;
- se evita que parezcan formularios genericos.

### 7. Estilos compartidos de pase especifico

Archivo: `frontend/src/styles/tailwind.css`

Se agregaron hooks compartidos:

- `[data-sd-specific-strip]`
- `.sd-chat-stage-cues`
- `.inbox-action-strip`
- `.analysis-action-strip`
- `.web-lab-stage-tabs`
- `.scenario-decision-card`
- `.scenario-option-card`
- `.scenario-flow-rail`
- `.call-safety-strip`
- `.call-controls-bar[data-sd-call-control-dock='true']`

Resultado:

- cada canal gana affordances especificos;
- las tiras son compactas y no reintroducen saturacion;
- se reutilizan variables de categoria de F6.R8.

### 8. Guard de regresion

Archivo: `test/f6-specific-simulation-pass-smoke.test.js`

Protege:
