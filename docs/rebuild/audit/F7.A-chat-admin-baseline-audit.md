# F7.A - Chat + Admin baseline audit

Fecha: 2026-04-23
Commit base: `0287e06` (`docs(F6.K): closeout validation`)

## 1. Why F7 opens now

F6 closed the simulation family and the shared closeout wrapper. What remains as large legacy surface area is `ChatDrawer` and `AdminView`, both already wired into the shell and route orchestration. That makes F7 a good place to harden those two areas without reopening F1-F6.

This phase is not about redesigning the shell or the lessons again. It is about taking the last two legacy product surfaces and making them feel deliberate, accessible, and easier to scan.

## 2. Current state of ChatDrawer

`frontend/src/components/ChatDrawer.jsx` is about 250 lines and is already visually coherent, but it still behaves like a custom overlay instead of a fully disciplined shell surface.

### What is already good

- It is mounted from the shell, not from random view code.
- It already adapts to viewport size with compact and wide layouts.
- It uses the new foundation pieces (`SurfaceCard`, `Button`, `Badge`, `cn`).
- The conversation auto-scrolls and the drawer can be closed with Escape or by clicking the backdrop.
- The empty state gives starter prompts, which is useful for first-time use.

### What is still weak

- It does not reuse the existing `OverlayFrame` foundation even though that component already solves focus trap and focus restore.
- The drawer advertises `role="dialog"` and `aria-modal="true"`, but the focus story is still hand-rolled.
- There is no explicit `aria-labelledby` / `aria-describedby` wiring from a reusable overlay shell.
- The message list is just a container of articles; it does not read like a live conversation region.
- The content is rendered through `dangerouslySetInnerHTML`, so the trust boundary is implicit and should be rechecked.
- The composer and launcher are split across shell code and drawer code, which is workable but still a little fragile.

### Live debt

- Focus trap and focus restore.
- Live region semantics for new messages.
- Cleaner relationship between launcher, overlay shell, and composer state.
- A clearer mobile/desktop behavior contract for the drawer.

## 3. Current state of AdminView

`frontend/src/components/AdminView.jsx` is about 412 lines and already does real work, but it reads more like a dense analytics dashboard than a calm admin surface.

### What is already good

- It has a loading state, empty state, refresh action, export action, and a back action.
- It is already responsive, with compact cards on mobile and denser tables on wider screens.
- The view is data-driven and uses the admin analytics helpers instead of hardcoding everything in JSX.
- The page already separates overview cards, charts, tables, and summary cards.

### What is still weak

- The hero at the top tries to communicate too many things at once.
- The first screen has a lot of panels, charts, and explanatory copy competing for attention.
- The admin view still feels more like a dashboard wall than a guided analysis surface.
- The tables are useful, but the hierarchy between overview, chart, and table sections could be sharper.
- The mobile stack is understandable, but it can still feel long and repetitive.
- The section language is not yet as consistent as the rest of the rebuilt UI.

### Live debt

- Information architecture and prioritization.
- Better section framing, especially for overview vs. deep detail.
- Stronger empty/loading states that feel like intentional admin states.
- Possible reuse of `PanelHeader`, `InfoPanel`, and `WorkspaceLayout` to reduce bespoke markup.

## 4. Product decision already visible

There is one important gating choice in the current code:

- `navigationPolicy.js` sets `allowChat: false` for `admin`.
- `buildShellSlots.jsx` respects that decision and does not mount the chat drawer there.

That may be correct, but F7 should confirm it explicitly. If admin is meant to stay chat-free, that should be a deliberate product choice. If admin needs chat support during review, that is a shell-policy decision, not a view-level tweak.

`SessionBar.jsx` already holds the admin preview toggle and the main navigation controls, so F7 can reuse that surface if the admin experience needs stronger discoverability.

## 8. Risks

- Focus management bugs if the drawer stays custom instead of reusing `OverlayFrame`.
- Admin density can easily become a long, tiring scroll if we keep every metric equally visible.
- The chat/admin gating decision can drift if the shell policy and route policy are not updated together.
- It is easy to spill F7 into general CSS cleanup, but that belongs to the later cleanup phase.
- There is no direct F7 smoke coverage yet, so the first implementation pass should be guarded carefully.

## 9. Recommendation

The proposed structure is good as-is:

- F7.A - baseline audit
- F7.B - ChatDrawer hardening
- F7.C - AdminView density / IA pass
- F7.D - shell and policy integration
- F7.E - closeout validation

The only caution is to keep F7.D explicit, because the chat/admin decision lives partly in shell policy, not only in the two views.
