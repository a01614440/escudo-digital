# F6.A - Simulation baseline audit

Fecha: 2026-04-23
Commit base: `68e0c15` (`audit(F6.A): whatsapp chat baseline`)

Esta auditoria abre F6 como una fase global de experiencia de simulaciones. La auditoria anterior `F6.A-whatsapp-chat-baseline-audit.md` queda como insumo especifico de WhatsApp/chat, pero no alcanza para el nuevo objetivo: revisar toda la familia de simulaciones como la superficie critica del producto.

No se implementaron cambios. No se tocaron simulaciones, renderer, registry, hooks, backend, services, contracts, scoring ni estilos.

## 0. Auditoria breve del estado

F6 es la fase mas importante del rebuild porque aqui vive la experiencia que el usuario realmente recuerda: practicar, detectar senales, equivocarse, recibir feedback y sentir que la app esta simulando un riesgo real. Ya no basta con que el flujo funcione o con que el lesson shell libere espacio; cada simulacion necesita leer bien, dominar la pantalla, tener identidad propia y sentirse jugable.

El estado actual confirma una base funcional aprovechable:

- `ActivityRenderer.jsx` y `activityRegistry.js` siguen estables como contratos.
- F5 ya agrego bypass de chrome generico para actividades inmersivas.
- Existen tests de logica para call, inbox, web lab y scenario flow.
- WhatsApp/chat y CallSimulation ya tienen lenguaje visual propio parcial.

Pero la familia completa todavia no esta lista para demo fuerte:

- demasiadas superficies usan texto secundario claro, fondos lavados y chips de bajo contraste;
- la informacion de apoyo compite con la simulacion principal;
- varias simulaciones siguen pareciendo worksheet/dashboard mas que experiencia;
- SMS e Inbox comparten demasiada estructura;
- WebLab aun se siente laboratorio escolar mas que pagina clonada creible;
- ScenarioFlow se lee como checklist con bitacora, no como flujo narrativo;
- CallSimulation tiene identidad fuerte, pero demasiadas cards informativas simultaneas;
- `signalActivities.jsx`, `app.css` y `legacy.css` mantienen deuda legacy visible alrededor de WhatsApp, compare domains, signal hunt y clases compartidas.

## 1. Estado actual de las simulaciones

### Que esta bien

- La separacion funcional central esta sana: registry, renderer, scoring y services no necesitan abrirse para esta fase.
- `sharedActivityUi.jsx` ya reconoce actividades inmersivas (`sim_chat`, `inbox`, `web_lab`, `call_sim`, `scenario_flow`) y evita el chrome generico para ellas.
- `FeedbackPanel.jsx` fue compactado en F5 y puede servir como base comun, siempre que no robe protagonismo.
- `CallSimulationActivity.jsx` ya tiene una identidad de llamada clara, fases internas y utilidades testeadas.
- `InboxActivity.jsx`, `WebLabActivity.jsx` y `ScenarioFlowActivity.jsx` tienen utilidades testeadas que permiten refinar UI sin cambiar scoring.
- WhatsApp/chat ya tiene una capa activa `sd-chat-*` que puede rescatarse.

### Que esta mal

- La familia no tiene un sistema consistente de contraste, densidad textual y jerarquia de simulacion.
- Muchas superficies usan `text-sd-muted`, `bg-white/55`, `bg-white/60`, `bg-sd-accent-soft`, `bg-sky-50` o variaciones muy suaves que pierden presencia sobre blanco.
- La app conserva demasiados patrones visuales viejos: `option-grid`, `domain-btn`, `message-box`, `signal-list`, `review-grid`, `activity-actions`, `wa-*`, estilos de inbox/web/call en `app.css` y clases duplicadas en `legacy.css`.
- El stage inmersivo existe, pero varias simulaciones vuelven a comprimir el contenido con grids internos, asides persistentes o paneles de apoyo demasiado grandes.
- Las instrucciones, summaries, bitacoras y feedback aparecen demasiado visibles al mismo tiempo.
- La identidad por categoria es desigual: WhatsApp y Call tienen caracter, SMS/Inbox comparten demasiado, WebLab no parece pagina real, ScenarioFlow no se siente suficientemente narrativo.

### Que sigue flojo o incomodo

- El contenido principal no siempre domina; en varias simulaciones el usuario lee paneles antes de entender la accion central.
- La experiencia no siempre se siente fullscreen: hay shells liberados por F5, pero las simulaciones se auto-encajonan.
- El texto largo se acumula en `ActivitySummaryBar`, instrucciones, asides, listas de senales, feedback y completion.
- Los estados finales suelen caer en `review-grid` o bloques de resumen que se sienten separados de la practica.
- La calidad de presentacion todavia varia mucho entre canales.

### Que sigue danando la experiencia

- Contraste insuficiente en labels, copy secundaria, chips y estados.
- Sobrecarga textual que reduce jugabilidad.
- Poca diferenciacion entre SMS e Inbox.
- WebLab con apariencia de worksheet y no de pagina clonada premium.
- WhatsApp/chat con implementacion hibrida: UI activa `sd-chat-*` y bloque legacy `wa-*` inaccesible dentro del mismo componente.
- Riesgo de tocar logica sensible por accidente si F6 intenta arreglar visuales desde registry, renderer o services.

## 3. Diagnostico por categoria

### WhatsApp / chat

Fuente principal: `frontend/src/components/activities/signalActivities.jsx`

Estado: tiene una base nueva `sd-chat-*`, pero sigue hibrida. El retorno activo ya parece mensajeria mas moderna, pero el componente conserva un segundo retorno legacy `wa-*` inaccesible, imports muertos y deuda de estilos heredados.

Problemas principales:

- `sd-chat-sim` sigue limitado a un ancho que impide una experiencia desktop realmente amplia.
- El thread no siempre se siente como superficie principal fullscreen.
- El composer no esta blindado como accion primaria pegada al flujo en todos los shells.
- El inspector/insight puede sentirse como texto de apoyo mas que lectura de riesgo integrada.
- Hay deuda a11y/copy/legacy documentada en la auditoria WhatsApp especifica.

Direccion:

- Mantener la base `sd-chat-*`.
- Eliminar hibridacion legacy en la subfase WhatsApp.
- Hacer thread/composer/feedback mas dominantes y menos dashboard.
- No tocar `requestSimulationTurn`.

### SMS

Fuente principal: `frontend/src/components/activities/immersive/InboxActivity.jsx` con `kind === 'sms'`

Estado: usa la misma base que correo/inbox. La diferencia es insuficiente para sentirse SMS real.

Problemas principales:

- Demasiado parecido a correo.
- Demasiada estructura de lista/detalle para un canal que deberia sentirse breve, austero y urgente.
- Labels y superficies suaves reducen presencia.
- El texto de apoyo puede competir con el mensaje.

Direccion:

- Separar presentacion por `kind` sin cambiar utilidades ni scoring.
- SMS debe sentirse mas seco, directo, de lectura rapida.
- Priorizar remitente, mensaje, accion y senales minimas.

### Inbox / email

Fuente principal: `frontend/src/components/activities/immersive/InboxActivity.jsx`

Estado: funcionalmente claro, pero todavia se siente panelizado. El cuerpo del correo, clasificacion, detalles y feedback compiten entre si.

Problemas principales:

- Lista + detalle existe, pero no alcanza a parecer cliente de correo premium.
- El inspector y la clasificacion viven demasiado cerca del cuerpo.
- Demasiados badges/pills/metadata a la vez.
- Review/completion se siente como formulario posterior.

Direccion:

- Convertirlo en cliente de correo con lectura principal y apoyo lateral/controlado.
- Sacar senales del cuerpo del correo cuando contaminen la ilusion.
- Mantener list/detail real en tablet/desktop y pantallas separadas en mobile.

### Web / WebLab

Fuente principal: `frontend/src/components/activities/immersive/WebLabActivity.jsx`

Estado: es la superficie con mayor exigencia visual y todavia no parece una pagina clonada creible. Se siente laboratorio educativo con hints, cards y grids.

Problemas principales:

- Page clone poco convincente.
- Pistas y paneles aparecen demasiado dentro de la experiencia.
- Grids internos y asides pueden deformar el contenido.
- Colores suaves y muchos labels reducen realismo.
- La practica no impresiona lo suficiente para demo.

Direccion:

- Hacer que la pagina falsa sea la protagonista.
- Mover inspector/senales fuera del clone.
- Evitar pistas obvias dentro de la pagina.
- Tratar WebLab como superficie premium, no como ejercicio escolar.

### Llamadas / CallSimulation

Fuente principal: `frontend/src/components/activities/CallSimulationActivity.jsx`

Estado: tiene identidad fuerte, fases reales y manejo funcional sensible. Es la simulacion con mas complejidad tecnica.

Problemas principales:

- Demasiadas cards informativas simultaneas en incoming, permissions y active.
- La experiencia puede sentirse panelizada pese a tener identidad de llamada.
- Voice/mic/browser APIs hacen riesgoso tocar comportamiento sin tests.
- En desktop existe riesgo de maqueta de telefono demasiado centrada o acompanada de exceso de informacion.

Direccion:

- Mantener la llamada como objeto principal.
- Hacer transcript, permisos y ayuda secundarios.
- No tocar backend ni servicios de IA/audio.
- Endurecer UI con smoke tests antes de pulir.

### ScenarioFlow

Fuente principal: `frontend/src/components/activities/immersive/ScenarioFlowActivity.jsx`

Estado: existe como flujo secuencial, pero visualmente se parece demasiado a worksheet con asides permanentes.

Problemas principales:

- La bitacora y rutina segura pueden competir con la decision.
- La historia no domina.
- El branching no se siente suficientemente narrativo.
- Completion/review cae en grid educativo.

Direccion:

- Tratarlo en F6.I como "other simulation refine" si se mantiene la estructura B-K.
- Hacer que la decision actual sea el foco.
- Bitacora y ayuda deben ser secundarias o colapsables segun shell.

### Comparacion / analisis y guided activities

Fuentes principales:

- `frontend/src/components/activities/signalActivities.jsx` (`CompareDomainsActivity`, `SignalHuntActivity`)
- `frontend/src/components/activities/basicActivities.jsx`

Estado: no son simulaciones inmersivas principales, pero si afectan la familia de practicas. Siguen usando clases legacy (`option-grid`, `domain-btn`, `signal-list`, `.btn`, controles nativos).

Problemas principales:

- Identidad analitica poco diferenciada.
- Contraste y estilo dependen de app/legacy CSS.
- Pueden quedar visualmente por debajo del estandar de F6 si se ignoran.

Direccion:

- Mantenerlas fuera del core de F6.B-H salvo contraste global.
- Incluirlas en F6.I/F6.J como "other simulation/activity surfaces" solo si el usuario autoriza alcance.
- No dejar que estas piezas expandan F6 hasta convertirse en F8 cleanup.

## 4. Relacion simulacion vs texto

El patron actual tiende a poner demasiado texto alrededor de la accion. El usuario llega a una simulacion y encuentra summary, instrucciones, labels, ayudas, asides, feedback y completion compitiendo. Eso reduce la sensacion de juego.

Regla para F6:

1. La simulacion domina.
2. La instruccion inicial debe ser corta.
3. La ayuda vive en disclosure, aside secundario o inspector segun shell.
4. El feedback debe aparecer cuando aporta, no como panel permanente gigante.
5. El cierre debe confirmar aprendizaje y siguiente accion sin abrir una pagina de reporte pesada.

## 5. Revision funcional

Sin ejecutar simulaciones ni modificar codigo, la lectura confirma estos puntos:

- Apertura: las simulaciones principales se resuelven desde `activityRegistry.js` y `ActivityRenderer.jsx`; estos contratos deben quedar congelados.
- Feedback: las actividades tienen mecanismos de feedback y completion, pero su presentacion puede robar espacio.
- Completion: existe en las familias principales, pero no siempre se siente integrado al flujo de practica.
- Navegacion: F5 ya libero el contenedor lesson; F6 debe evitar reintroducir encajonamiento desde layouts internos.
- IA/audio/timers: WhatsApp y Call usan servicios/comportamientos sensibles; WebLab usa countdown y estados internos. Cualquier refinamiento visual necesita tests de no regresion.

Cobertura actual detectada:

- `test/call-simulation.test.js`: utilidades de llamada.
- `test/immersive-logic.test.js`: normalizadores y scoring logico de inbox, weblab y scenario.
- Tests F5 protegen shell/chrome, no calidad interna de simulaciones.

Hueco de cobertura F6:

- No hay suficientes smoke tests estructurales por categoria.
- No hay guardas para stage dominance, identidad por categoria, ausencia de clases legacy en superficies migradas o reduccion de texto visible.

## 6. Presentation quality / demo-readiness

Hoy las simulaciones no fallan por una sola cosa grande, sino por acumulacion:

- mucho texto alrededor de la accion;
- poco contraste en elementos secundarios;
- canales que se parecen demasiado;
- panels y grids que recuerdan a dashboard;
- stage que se siente liberado por F5, pero re-encajonado por la simulacion;
- WebLab y ScenarioFlow todavia no llegan al nivel visual necesario para impresionar.

Para demo, F6 debe hacer que cada categoria sea reconocible en segundos:

- WhatsApp/chat: conversacion viva, thread y composer dominantes.
- SMS: lectura rapida, urgente, austera.
- Inbox/email: cliente de correo real, lectura limpia, inspector externo.
- WebLab: pagina clonada creible, no maqueta escolar.
- Calls: llamada principal, audio/transcript como apoyo.
- Scenario/analysis: decision narrativa o analitica, no checklist plana.

## 9. Riesgos

### Visuales

- Resolver contraste solo con colores mas fuertes puede destruir la identidad calmada del producto si no se hace por categoria.
- Convertir todas las simulaciones en fullscreen sin reglas puede ocultar contexto util.
- Reusar el mismo lenguaje visual en todas las categorias dejaria F6 tecnicamente limpia pero visualmente plana.

### Funcionales

- WhatsApp/chat y CallSimulation dependen de servicios y browser APIs; no deben refactorizarse desde logica.
- WebLab tiene timers/estado interno; cualquier cambio de layout debe preservar stages, countdown y completion.
- Inbox/SMS comparten componente y utilidades; separar presentacion no debe separar scoring.

### Scope creep

- F6 puede convertirse facilmente en cleanup total de `app.css`/`legacy.css`. Eso no corresponde salvo clases usadas por la simulacion que se esta cerrando.
- Actividades basicas y analiticas pueden crecer el alcance. Deben entrar solo en F6.I/F6.J si el usuario lo autoriza.
- No se debe volver a F5 para redisenar LessonView salvo bloqueo demostrado.

### Mezcla shell vs simulacion

- F5 libero el contenedor; F6 debe corregir layout interno de simulaciones.
- Si una simulacion sigue aplastada, primero revisar su grid/asides/width internos antes de culpar al lesson shell.
- `ActivityRenderer` y `activityRegistry` no son lugares para polish visual.

## 10. Recomendacion final

La estructura B-K propuesta se mantiene, con un ajuste minimo:

- F6.A queda como auditoria global de simulaciones.
- F6.B, F6.C y F6.D deben ejecutarse antes de refinar canales individuales porque corrigen problemas sistemicos: contraste, densidad textual, identidad por categoria y dominancia de stage.
- F6.E-F6.H deben cerrar WhatsApp, SMS, Inbox y WebLab como canales principales.
- F6.I debe llamarse y entenderse como `Calls / ScenarioFlow / other simulation refine`, porque ScenarioFlow no debe quedar perdido y las superficies de analisis necesitan una decision de alcance.
- F6.J debe ser un pase transversal de funcionalidad/polish, no una fase para inventar nuevas simulaciones.
- F6.K debe validar con tests, build, build-storybook, smoke manual y cierre binario.

No conviene volver a F6.A.1 WhatsApp-only como siguiente paso. Ese plan quedo superado por la decision humana de abrir F6 global. El siguiente paso correcto es:

**F6.B - Contrast / readability / text-density system pass.**

## 11. Cierre de F6.A

Esta auditoria confirma que el problema central de F6 no es solo "hacer bonitos" los canales. El problema real es que la experiencia de simulacion todavia mezcla demasiada ayuda visible, contraste flojo, identidad desigual y layouts internos que vuelven a comprimir el contenido despues de que F5 ya habia liberado el lesson shell.

F6 debe cerrar la familia completa con una regla simple:

**la simulacion domina, el texto ayuda, el chrome no ahoga.**

F6.A queda lista para pedir autorizacion humana de F6.B.
