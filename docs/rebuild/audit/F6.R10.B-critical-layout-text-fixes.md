# F6.R10.B - Critical layout + text fixes

Fecha: 2026-04-24

## 1. Auditoria breve

F6.R10.A confirmo que el problema urgente no era solo de simulaciones: `Mi ruta` seguia siendo la superficie mas rota visualmente. La captura real mostraba texto que solo se distinguia al seleccionarlo, bloques flotando, jerarquia partida, exceso de columnas y alto riesgo de titulos partidos verticalmente.

Esta subfase ataco los fixes criticos de layout y texto que bloqueaban una presentacion decente:

- `Mi ruta` no podia seguir usando una superficie inversa/lavada para copy oscuro sobre fondo claro.
- La continuidad, progreso y CTA tenian que vivir en un solo bloque fuerte, no dispersos.
- El navegador de modulos no podia seguir comportandose como rail angosto/sticky si eso rompia lectura.
- Los nombres de modulos y actividades necesitaban clamps normales, sin `break-words` ni word splitting vertical.
- La pantalla de ruta debia dejar de forzar columnas desktop cuando el contenido necesitaba respirar.

### Decisiones principales

- `RouteBriefing` cambio a `variant="editorial"` y `data-sd-route-shelf="critical-recovery"` para abandonar la mezcla visual de superficie inversa con texto dificil de leer.
- La continuidad, el porcentaje, el progreso y el CTA principal quedaron integrados en `.sd-route-continuity-card`.
- `DashboardSceneBar` paso a una superficie clara y legible para evitar bloques oscuros/desconectados dentro de la ruta.
- `RouteNavigatorRail` dejo de ser rail sticky estrecho y paso a bloque secundario ancho con `data-sd-route-rail="secondary-block"`.
- La lista de modulos ahora usa una grilla amplia `sm:grid-cols-2 xl:grid-cols-3` dentro del bloque secundario, en vez de estrechar titulos.
- El cuerpo de ruta cambio a `data-sd-route-comfort="stacked-recovery"` para no forzar una columna de detalle junto al navegador.
- Los titulos de modulos y actividades usan clases dedicadas (`sd-route-pill-title`, `sd-module-activity-title`) con clamp de dos lineas, `word-break: normal`, `overflow-wrap: normal` y `hyphens: none`.

## 3. Correcciones por frente

### Mi ruta

- Se retiro el tono inverso del hero de ruta.
- Se subio contraste de titulo, subtitulos, labels, badges y copy.
- Se unifico continuidad + progreso + CTA en una sola tarjeta interna.
- El CTA principal queda anclado dentro del bloque de siguiente paso.

### Columnas / rails

- El rail de ruta dejo de ser sticky y angosto.
- La ruta ya no fuerza el layout desktop de detalle + rail.
- El navegador ahora aparece como bloque subordinado ancho, con modulos distribuidos sin aplastar texto.

### Texto partido verticalmente

- Se retiraron expectativas y clases `break-words` en titulos criticos.
- Se agregaron clamps de dos lineas para nombres de modulo y actividad.
- Se reforzaron reglas CSS para evitar `word-break` agresivo, hyphenation y wraps letra por letra.

### Contraste

- Se agrego una capa CSS especifica para `critical-recovery`.
- Se endurecio el color de copy dentro de route briefing, continuity card, scene bar, navigator block y mission board.
- Se reforzaron badges `soft` y `accent` en route/navigator/detail.

### Simulaciones, feedback y completion

- No se tocaron en F6.R10.B. Esta subfase fue el fix critico de layout/texto detectado en `Mi ruta`; las simulaciones especificas siguen protegidas por los cambios y guards existentes de F6.R10 previo.

## 5. Validacion visual/manual

No se genero una captura automatizada nueva en esta subfase. La validacion visual directa queda cubierta por los guards estructurales agregados para los problemas observados en la captura:

- `RouteBriefing` ya no usa superficie inversa ni texto inverso en `Mi ruta`.
- `data-sd-route-shelf="critical-recovery"` activa la capa de contraste para el hero.
- `data-sd-route-comfort="stacked-recovery"` evita columnas desktop forzadas en la ruta.
- `data-sd-route-rail="secondary-block"` confirma que el navegador ya no es un rail estrecho.
- `sd-route-pill-title` y `sd-module-activity-title` protegen contra palabras partidas verticalmente.

La captura real de presentacion debe repetirse despues de levantar la app para confirmar percepcion final en navegador, pero los puntos estructurales que causaban el fallo visible quedaron cerrados en codigo y tests.
