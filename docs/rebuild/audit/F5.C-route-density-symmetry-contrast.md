# F5.C — Route density / symmetry / contrast cleanup

Fase: F5.C.
Alcance: aplanar el cuerpo de la pestaña "Ruta" en `CoursesView` a dos paneles en desktop, eliminar la columna permanente de insight, compactar los pills de módulo, absorber metadata útil en el detalle, reducir duplicación en `ProgressScene`, colapsar `SettingsScene` a una sola superficie, y subir contraste en estados selected/next.

## Objetivo (del audit F5.A)

Resolver los problemas de matriz #5, #6, #7, #8, #9 y parcialmente #19:

- demasiadas columnas (workspace 3-col + grid 2-col anidado en `ModuleMissionBoard`);
- stats decorativas (`ModuleMissionBoard` StatStrip, `SettingsScene` StatStrip, `ProgressScene` ProgressSummary duplicada);
- contraste pobre (`bg-sd-accent-soft` como fondo dominante, Badge `soft` sin peso, metadata redundante en pills);
- ruta por módulos incómoda (pills demasiado cargados, activity list siempre visible);
- ajustes desproporcionado (SplitHeroLayout para 3 selects);
- journey stepper en scene bar que competía con el propio encabezado de escena.

Regla central aplicada: **content-first**. En la pestaña Ruta, el contenido principal ahora es rail de módulos + detalle. Todo lo que antes era "insight lateral" se absorbió donde aporta decisión o se eliminó.

## Hallazgos

### Lo que sí cambia visualmente
- La pestaña "Ruta" deja de sentirse como un dashboard con 3 columnas. Es rail + detalle.
- Cada pill de módulo es una tarjeta compacta con 3 datos reales (index, título, estado) + progreso. Sin metadata decorativa.
- El detalle del módulo tiene CTA primero, progreso, metadata útil (gap + siguiente desbloqueo), y un toggle opcional para las actividades — no las fuerza.
- Ajustes ya no se siente como "otra pantalla". Es un form compacto.
- El snapshot de historial en progreso ahora es una fila limpia, no una tarjeta translucida.

### Lo que no cambió (fuera de scope)
- `LessonView.jsx` sin tocar: scope F5.D+.
- `DashboardSceneBar` mantiene su estructura (sólo dejamos de pasarle `journeySteps`).
- `FeedbackPanel.jsx`, `sharedActivityUi.jsx`, `ActivityRenderer.jsx`, `activityRegistry.js`: no tocados.
- `StatStrip`/`ProgressSummary`/`PanelHeader` patterns: no tocados.
- `tokens.js`: no tocado.

### Riesgos residuales para próximas subfases
- El `<details>` de las actividades del módulo necesita validación de UX real: si el usuario confía en "ver la lista" antes de abrir, podría abrirlo al seleccionar el módulo. Se revisa en F5.G si aplica.
- El contraste de `Badge tone="soft"` ("Modo admin") sobre superficies inverse sigue siendo flojo; se atacará en F5.G con el pase global de contraste.
- La clase `sd-route-pill` y `sd-module-mission-board` / `sd-module-activities-toggle` / `sd-settings-scene` / `sd-module-activity-list` quedan como hooks sin CSS nuevo (reutilizan tokens existentes). Si F5.G necesita stylings específicos, ya tienen selector estable.
- Al ir a 2-col en desktop, el rail pierde su `sticky` antiguo (`xl:sticky xl:top-6` sigue adentro del SurfaceCard del RouteNavigatorRail pero el wrapper externo ya no lo gate-ea). Validar scroll en desktop largo en F5.G.
