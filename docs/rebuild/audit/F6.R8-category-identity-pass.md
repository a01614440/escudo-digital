# F6.R8 - Category identity pass

Fecha: 2026-04-24
Estado: cerrada localmente

## Objetivo

Hacer que cada familia de simulacion tenga una identidad visual mas clara y menos generica sin reabrir logica, scoring, backend ni contratos de renderer.

F6.C ya habia creado una base tecnica de categorias. F6.R8 eleva esa base a lenguaje visible de producto: firma, ritmo, cue visual y tokens compartidos por categoria.

## Cambios implementados

### 1. Metadata de identidad por categoria

Archivo: `frontend/src/components/activities/immersive/shared.js`

Se amplio `SIMULATION_CATEGORY_META` con:

- `signature`
- `cue`
- `rhythm`

Categorias cubiertas:

- `chat`
- `sms`
- `email`
- `web`
- `call`
- `scenario`
- `analysis`
- `concept`

Tambien se agrego `getSimulationCategoryMeta(category)` y se ajusto el fallback para actividades guiadas:

- `concepto`
- `quiz`
- `checklist`
- `abierta`
- `simulacion`

ahora caen en la categoria visual `concept`.

Resultado:

- las actividades guiadas de concepto/habito dejan de caer en una identidad generica de analisis;
- cada categoria tiene un lenguaje minimo propio para UI visible.

### 2. Banda visual de identidad compartida

Archivo: `frontend/src/components/activities/sharedActivityUi.jsx`

Se agrego `SimulationIdentityBand`.

La banda expone:

- `sd-simulation-identity-band`
- `data-sd-category-channel`
- `data-sd-category-rhythm`
- `sd-simulation-identity-dot`
- `sd-simulation-identity-cue`

Se renderiza en:

- `ActivityChrome` inmersivo como banda compacta antes del stage;
- `ActivityChrome` guiado dentro del briefing.

Resultado:

- cada practica comunica su canal antes de interactuar;
- la identidad se vuelve transversal sin meter copy largo;
- se conserva la reduccion de densidad de F6.R7.

### 3. Tokens visuales de firma por categoria

Archivo: `frontend/src/styles/tailwind.css`

Se agregaron tokens compartidos:

- `--sd-simulation-category-ink`
- `--sd-simulation-category-rail`
- `--sd-simulation-category-pattern`

Se reforzaron las categorias:

- chat: conversacional / verde mensajeria
- sms: austera / mensaje corto
- email: bandeja / azul de correo
- web: laboratorio web / grilla de navegador
- call: urgencia / pulso de llamada
- scenario: flujo / decisiones encadenadas
- analysis: criterio / evidencia
- concept: habito / estudio guiado

Resultado:

- las superficies ya no dependen solo de badges;
- cada familia deja un rastro visual distinto en fondo, borde, rail, dot y acentos.

### 4. Consumo real en simulaciones mayores

Archivo: `frontend/src/styles/tailwind.css`

Se conectaron variables de categoria a superficies existentes:

- WhatsApp/chat:
  - burbuja del usuario usa `--sd-simulation-category-rail`;
  - avatar y sugerencias usan acento de categoria;
  - superficie del chat usa glow/borde de categoria.
- SMS/email:
  - shell de inbox usa fondo/patron de categoria;
  - headers y items activos consumen borde/acento de categoria.
- WebLab:
  - shell principal usa patron web;
  - browser bar y CTA consumen acento web.
- Calls:
  - phone shell, status/topbar, avatar y ondas usan acento call.
- Scenario:
  - botones hover y superficies principales usan acento scenario.
- Concept/analysis:
  - tarjetas de concepto, opciones, dominio y signal chips consumen identidad propia.

Resultado:

- las simulaciones se diferencian mas por categoria;
- se reduce la sensacion de "misma UI blanca con distinto texto";
- la identidad se aplica en componentes compartidos y en piezas especificas.

### 5. Guard nuevo de regresion

Archivo: `test/f6-category-identity-pass-smoke.test.js`

Protege:

- metadata de firma por categoria;
- `SimulationIdentityBand`;
- tokens `ink`, `rail` y `pattern`;
- categoria `concept`;
- consumo real de variables en chat, SMS, email, WebLab, calls, concept y analysis.
