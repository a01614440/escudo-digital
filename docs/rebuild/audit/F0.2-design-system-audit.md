## 0. Nota inicial sobre "deep research report"

El prompt de F0.2 pide comparar contra "lo recomendado en el deep research report (color semántico por contexto)". En el repo no hay un documento con ese nombre ni fuera de `docs/rebuild/`. Lo más cercano es `frontend/src/design-system/README.md`, que funciona como doctrina visual interna. La comparación se hace contra:

1. Ese README interno.
2. El framework estándar de **color semántico educativo** (éxito / alerta / riesgo / simulación / progreso), que es el que el prompt de F0 pide explícitamente evaluar.

Si existe un deep research report externo que no está en el repo, este audit no pudo considerarlo. **Riesgo informativo** — no bloquea F0.2.

## 1. Cobertura de tokens — visión general

Los tokens viven **duplicados** en dos sitios:

- **`frontend/src/design-system/tokens.js`** — fuente JS (objetos exportados). Consumido por tests (`foundation-smoke.test.js`) y por **nadie más** del runtime (grep: 0 imports desde JSX/JS fuera de tests).
- **`frontend/src/styles/tailwind.css`** — CSS custom properties (`--sd-*`) declaradas en `:root`, `body[data-theme='dark']`, y expuestas a Tailwind vía `@theme inline` como `--color-sd-*`. Es **el consumidor real** del runtime.

**Riesgo estructural:** dos fuentes de verdad que tienen que mantenerse sincronizadas a mano. El test `foundation-smoke.test.js` verifica algunas claves de `tokens.js`, pero **no verifica equivalencia con los valores de `tailwind.css`**. Si alguien cambia un valor en un lado y no en el otro, nada lo detecta.

### 2.3 Typography — 🔴 **lagunar**
- En `tokens.js`: **nada**. No hay `typographyTokens`, `fontTokens`, `textTokens`.
- En `tailwind.css`: solo 2 tokens:
  - `--font-sans: "Sora", …` (línea 194)
  - `--font-display: "Unbounded", …` (línea 195)
- Escala tipográfica: **vive solo en clases de componente** dentro de `@layer components` (`.sd-title-display`, `.sd-title`, `.sd-heading-md`, `.sd-heading-sm`, `.sd-subtitle`, `.sd-copy-sm`, `.sd-copy`, `.sd-copy-strong`, `.sd-copy-inverse`, `.sd-eyebrow`, `.sd-overline`, `.sd-label`, `.sd-caption`). Tamaños (`text-[2rem]`, `text-xl`, `text-sm`), line-heights (`leading-[0.98]`, `leading-7`, `leading-6`) y letter-spacing (`tracking-[-0.06em]`, `tracking-[0.06em]`) están incrustados como **valores literales dentro de los `@apply`**, no expuestos como tokens.

**Consecuencia práctica:** cualquier componente nuevo que necesite "texto con estilo título medio" tiene que recrear el heading copiando los valores o depender del selector `.sd-heading-md`. No hay forma de exponer estos valores a JS ni a tests.

## 3. Tokens faltantes según el playbook y el marco educativo

### 3.1 Lo que el README interno ya enumera vs. lo que falta

El README de `design-system/` enumera tokens normativos y lo que **debería** estar presente. Cotejo:

| Requisito del README | ¿Existe en `tokens.js`? | ¿Existe en `tailwind.css`? |
|---|---|---|
| canvas | ✅ | ✅ |
| surface base | ✅ | ✅ |
| region surface (hero/command/support/insight/spotlight/inverse) | ✅ | ✅ |
| border | ✅ | ✅ |
| text | ✅ | ✅ |
| accent | ✅ | ✅ |
| semantic (success/warning/danger soft) | ✅ | ✅ |
| spacing scale 1-11 + region gaps + shell spacing | ✅ | ✅ |
| radius xs-xl + pill | ✅ | ✅ |
| shadow xs/sm/panel/md/lg/spotlight/floating | ✅ | ✅ |
| blur soft/panel/hero | ✅ | ✅ |
| motion fast/base/slow/enter + standard/emphasis/decisive | ✅ | ✅ |
| z-index base…overlay | ✅ | ✅ |

El README **no menciona typography como requisito** — tampoco se autoexige. Pero el prompt de F0 sí la pide como categoría. El gap es real.

El README menciona también `editorial` como región semántica, pero **no existe `--sd-surface-editorial`** como token propio. Se resuelve inline dentro de `.sd-region-editorial` con `rgba(250,252,255,0.98)` hardcoded (línea 411 de tailwind.css). 🔴 inconsistencia.

### 3.2 Lo que el marco educativo pide y que falta

El prompt enumera tokens educativos: **éxito, alerta, riesgo, simulación, progreso**.

| Categoría educativa | Estado actual | Hallazgo |
|---|---|---|
| **éxito** | ✅ `--sd-success` + `--sd-success-soft` | Cubierto. La doctrina (`interactionRules.success`) dice reservarlo para confirmación y avance, no como CTA primario — correcto. |
| **alerta** | ✅ `--sd-warning` + `--sd-warning-soft` | Cubierto. Usado en `.sd-inline-message-warning`, `.sd-badge-warning`, `.sd-region-spotlight`. |
| **riesgo** | ✅ `--sd-danger` + `--sd-danger-soft` | Cubierto. Pero: el producto es educativo contra phishing. **No hay distinción entre "error de usuario" (ej. respuesta incorrecta) y "señal de peligro" (ej. evidencia de estafa)**. Ambos colapsan al mismo token `danger`. Semánticamente son dos contextos distintos que el usuario lee diferente. 🔶 extender. |
| **simulación** | 🔴 **ausente** | No hay `--sd-surface-simulation` ni un token específico para contextos inmersivos (WhatsApp, SMS, Inbox, Call, WebLab). Cada actividad inmersiva improvisa: `signalActivities` usa `#0e172a`/`#0b1221` hardcoded en chat header; `InboxActivity`, `WebLabActivity` usan `bg-white/75` + `rgba(15,23,42,0.45)`. El shell contract (`docs/rebuild/05`) exige que las simulaciones sean "superficies protagonistas" — esto necesita tokens propios. 🔴 rehacer para F6. |
| **progreso** | 🔶 mezclado | Los tokens de progreso colapsan a `--sd-accent` (el gradient principal de `.sd-progress-bar`) y opcionalmente a `success`/`warning`. **No hay un token semántico "progress"** separado del accent primario. Consecuencia: cualquier cambio de accent rompe progreso, y cualquier UI secundaria con accent se confunde con "progreso". 🔶 extender. |

### 3.3 Tokens educativos que no están pero convendría considerar

Para un producto de formación en ciberseguridad:

- `--sd-evidence` / `--sd-evidence-soft` — señal detectada / indicador sospechoso en texto. Hoy se resuelve con `danger`, pero evidencia ≠ error.
- `--sd-safe-action` / `--sd-safe-action-soft` — acción segura recomendada. Hoy se resuelve con `success`, pero acción segura ≠ usuario acertó.
- `--sd-streak` / `--sd-streak-contrast` — logros, rachas, hitos. Hoy no existe.
- `--sd-reflection` / `--sd-reflection-soft` — feedback de coach, post-mortem, bitácora. Hoy se resuelve con `insight`, pero es un caso más específico.
- `--sd-simulation-dark` / `--sd-simulation-neutral` / `--sd-simulation-warm` — superficies inmersivas por canal (chat dark, email neutral, call dark). Imprescindible para F6.
- Progreso específico: `--sd-progress-track` (ya existe como mix, no como token), `--sd-progress-fill`, `--sd-progress-fill-complete`.

## 5. Consistencia de nombres y escalas

### 5.1 Consistencia entre `tokens.js` y `tailwind.css`

| Aspecto | JS (tokens.js) | CSS (tailwind.css) | Drift |
|---|---|---|---|
| colorTokens.canvas | array de nombres | variables con valores | valores en un solo lado |
| colorTokens.semantic | array (`--sd-success`, `--sd-success-soft`, …) | variables con valores | ídem |
| overlay | **no existe** en tokens.js | `--sd-overlay` sí existe | 🔴 inconsistencia |
| radiusTokens.xs-pill | objeto con valores | `--sd-radius-*` con valores | valores duplicados, riesgo drift |
| spacingTokens.scale | objeto numérico `1..11` | `--sd-space-1..11` | valores duplicados |
| shellSpacingTokens | objeto profundo mobile/tablet/desktop/widths | 21 variables `--sd-shell-*` con mismos valores | valores duplicados |
| shadowTokens | objeto con valores | `--sd-shadow-*` con mismos valores | ídem |
| blurTokens | objeto | `--sd-blur-*` | ídem |
| motionTokens | duraciones + curvas | `--sd-motion-*` + `--sd-ease-*` | ídem |
| zIndexTokens | numéricos | `--sd-z-*` | ídem |
| **typography** | ❌ no declarado | solo `--font-sans` + `--font-display` | 🔴 ausente en ambos lados |

**Consumo JS real:** grep de `from.*design-system` o `from.*tokens\.js` en `frontend/src/` devuelve **0 imports** fuera de `test/foundation-smoke.test.js`. Es decir: **el módulo JS del design-system es inerte en runtime**. Solo existe para los tests.

Esto convierte a `tokens.js` en un **contract mirror sin enforcement**. Correcciones posibles:
- Invertir la dirección: generar `tailwind.css` desde `tokens.js` (build step).
- O generar `tokens.js` desde `tailwind.css` (parser).
- O añadir tests adicionales que verifiquen equivalencia de valores.

| Patrón | Consistencia |

| z-index (gap 20/20/20/10/5/5/10/10) | base=0, sticky=20, drop=40, drawer=50, scrim=55, modal=60, toast=70, overlay=80 | 🔶 gaps desiguales dificultan inserciones futuras |

| Respuesta incorrecta del alumno | `--sd-danger` | 🔶 colapsa con "amenaza detectada" |

### Veredicto global del design system: 🔶 **EXTENDER**

- No hay que rehacer el sistema — la arquitectura, nomenclatura y cobertura base son sólidas y ya están testadas.
- Pero antes de F1 quedar "cerrada", hay **cuatro extensiones no-opcionales**:
  1. Tokens de **typography** completos (type-scale, weights, line-heights, letter-spacing, font-families) expuestos como CSS vars y en `tokens.js`.
  2. Tokens educativos extendidos (**evidence**, **safe-action**, **streak**, **simulation-{dark|neutral|warm}**, **progress-{track|fill|fill-complete}**).
  3. Alineación `tokens.js` ↔ `tailwind.css`: o bien generación automática, o bien test de equivalencia. Hoy hay **drift sin detección** (ej. `--sd-overlay` existe en CSS y no en JS).
  4. Eliminación de hex/rgba hardcoded en JSX (**~13 ocurrencias de sombras + 1 gradient hero + 64 literals `white`**) sustituyéndolos por clases tokenizadas o por tokens nuevos.

## 8. Riesgos para fases siguientes

1. **F1 aún no está cerrada** aunque la doctrina y la estructura lo sugieran. Typography sin tokens es un gap grande que va a multiplicarse con cada vista nueva.
2. **F5 y F6 van a exponer la laguna educativa del color**. Inspector + evidence + safe-action + immersive surfaces no tienen tokens; los agentes van a volver a improvisar hex como ya lo hicieron en `WebLabActivity.jsx` (18 arbitrary-value classes) y `signalActivities`.
3. **El drift entre `tokens.js` y `tailwind.css`** es silencioso. Si alguien cambia un valor en un solo lado, los tests pasan y la app se ve bien en modo claro pero mal en dark (o viceversa).
4. La regla "no más `text-white`/`bg-white` directos" **no está documentada ni linteada**. 64 ocurrencias en 13 archivos dicen que se aplicó ad libitum.
5. El **hardcoded hero** `#0a1d35/#123052` en `CoursesView.jsx:599` es bandera de F4 migrado sin cerrar la foundation subyacente.

## 9. Micro-hallazgos (notas, no bloqueantes)

- `.sd-lesson-mission-hero` en `tailwind.css:924-977` tiene **más de 50 líneas con `!important`** para forzar estilos sobre primitives. Es un code smell — si un hero necesita tanto override, o el hero no es primitive o el override es foundation.
- `.sd-badge-soft` (línea 593-598) está declarado pero no aparece en la documentación del README.
- `containerAwarenessRules.requiredFor` lista 7 componentes pero `data-sd-container="true"` solo se aplica automáticamente vía CSS `[data-sd-container='true']` — depende de que cada componente añada el atributo. No hay test que verifique que los 7 lo añaden.
- `foundationInventory.primitives` en `tokens.js` lista 15 primitives, `foundationInventory.patterns` lista 10, `foundationInventory.layouts` lista 2. El test `foundation-smoke` solo verifica presencia parcial (StageHero, SupportRail, ActionCluster, PanelHeader, SplitHeroLayout, WorkspaceLayout). Los demás no tienen verificación.
- El tema `dark` en `body[data-theme='dark']` está definido pero **no se aplica automáticamente** — depende de `.storybook/preview.jsx` que lo escribe manualmente; en la app real no hay toggle ni media-query. ¿Es dead code? A verificar en F0.4.

## 10. Recomendación específica antes de abrir F1.X

Si F1 se va a dar por cerrada formalmente después de F0, hay **tres pasos mínimos** que caben dentro del alcance de F1 (foundation) y evitan arrastrar deuda a F2–F6:

1. Añadir `typographyTokens` a `tokens.js` y `--sd-text-size-*`, `--sd-line-height-*`, `--sd-font-weight-*`, `--sd-letter-spacing-*` a `tailwind.css`. Reescribir `.sd-title-display`, `.sd-heading-*`, `.sd-copy*`, `.sd-eyebrow`, etc. usando esos tokens.
2. Añadir tokens educativos extendidos (**evidence**, **safe-action**, **streak**, **simulation-{dark|neutral|warm}**, **progress-{track|fill|complete}**).
3. Reemplazar las 12 sombras hardcoded y el 1 hex de `CoursesView` por tokens/clases del sistema. Sustituir los 64 `text-white`/`bg-white` por clases tokenizadas (`text-sd-text-inverse`, `bg-sd-surface`).

Si estas tres no se hacen, F1 **no debería considerarse cerrada** — la foundation va a fracturarse durante F5 y F6 y se rehará tarde a mayor costo.
