## 2. Punto único de importación

Ambos archivos se importan solo en **`frontend/src/main.jsx`**:

```js
// main.jsx líneas 5-6
import './styles/legacy.css';
import './styles/app.css';
```

No hay otros imports de estos archivos en el repo. Eliminar las dos líneas de `main.jsx` + los dos archivos es el único cambio necesario en la capa de imports — siempre que todos los componentes dependientes ya estén migrados.

## 3. Componentes que importan / dependen de legacy directamente

### 3.1 Dependencia de variables CSS

`app.css` usa **~50+ veces** las variables CSS de `legacy.css` (`--muted`, `--primary`, `--border`, `--ink`, `--success`, `--warning`, `--danger`, etc.). Esto significa que **si se elimina `legacy.css` sin eliminar `app.css` simultáneamente**, todas las reglas de `app.css` que usan esas variables perderían sus valores.

**Los dos archivos deben retirarse juntos, no por separado.**

Grep confirmó: el código JSX/JS **no usa ninguna variable de `legacy.css`** directamente — ni `var(--primary)`, ni `var(--ink)`, ni `var(--muted)`. El nuevo sistema usa `--sd-*`. La dependencia es enteramente CSS → CSS.

## 5. Dependencias cruzadas legacy ↔ nuevo

### 5.1 Cruzamiento de sistema de diseño

El sistema legacy define variables `--primary`, `--bg`, `--ink` en `legacy.css`. El nuevo sistema define `--sd-accent`, `--sd-canvas`, `--sd-text` en `tailwind.css`. **No hay cruzamiento real**: el código nuevo usa solo `--sd-*`; el código legacy usa solo `var(--primary)` etc. vía app.css.

**Excepción**: `tailwind.css` y `legacy.css` ambos definen estilos para `body` y `body[data-theme='dark']`. Hay **cascada competitiva**:
- `legacy.css:40-50` define `body { padding: 32px 20px 60px; background: ... }`.
- `tailwind.css:237-244` define `body { background: ...; color: var(--sd-text); font-family: ... }`.
- Tailwind carga después (ver main.jsx orden: legacy → app → tailwind), por lo que **tailwind.css gana** en la mayoría de propiedades. El `padding: 32px` de legacy.css queda sobreescrito.
- **Pero**: si legacy.css se elimina y hay elementos legacy que dependían de ese padding, podrían verse diferentes.

### 5.2 Cruzamiento de `sd-chat-*` en signalActivities.jsx

`signalActivities.jsx` es la única actividad del set que **mezcla clases legacy y nuevas**:
- Líneas 349-881: usa `sd-chat-activity-shell`, `sd-chat-sim`, `sd-chat-surface`, `sd-chat-header`, `sd-chat-thread`, `sd-chat-bubble`, `sd-chat-composer` — estas son **clases sd-* de tailwind.css**.
- Líneas 320-348, 485-531: usa `wa-side-panel`, `signal-list` — estas son **clases legacy de app.css**.
- **Conclusión**: signalActivities.jsx ya tiene parte de la migración a `sd-chat-*` iniciada. El bloque WhatsApp (6A) está a medio migrar — la UI visual del thread/compose ya usa sd-*, pero el panel lateral y los indicadores de señal usan legacy.

### 5.3 Cruzamiento de clases app-shell-* (ya analizado en F0.4)

Los tres shells (`MobileShell`, `TabletShell`, `DesktopShell`) generan clases:
- `app-shell` — definida en `app.css:97-104` (display grid, min-height, etc.).
- `app-shell-mobile`, `app-shell-family-mobile` — definidas en `app.css:5497-5500`.
- `app-shell-tablet`, `app-shell-desktop` — similares.

Estas clases tienen definiciones en app.css que los shells nuevos podrían necesitar como fallback. Verificar si eliminarlas rompe layout antes de borrar app.css.

### 5.4 `body[data-viewport]` — el acoplamiento más profundo

`app.css` tiene 64 reglas de tipo `body[data-viewport='phone-small'] .session-main`. Estas reglas estaban diseñadas para el sistema legacy de responsive.

**Estado actual**: `useResponsiveLayout.js` sigue escribiendo `body.dataset.viewport = profile.viewport` en cada resize. Por tanto, las 64 reglas siguen funcionando para **clases legacy** (`session-main`, `session-main-actions`, etc.). Al migrar estas clases (F7: SessionBar/ChatDrawer), las 64 reglas quedarán sin efecto.

**La eliminación de `body[data-viewport]` en app.css es dependiente de migrar los componentes que las usan (F7).**

## 7. Estimación de dificultad de eliminación

### legacy.css completo — **Alto**

Razones:
- Tiene 311 selectores. Ninguno lo consume el código nuevo (JSX) directamente, pero `app.css` depende de sus variables.
- La eliminación requiere eliminar `app.css` al mismo tiempo.
- Si se elimina antes de que `app.css` migre, las actividades legacy perderían sus estilos base.
- **Debe ser la última acción** de F8, después de que todos los componentes legacy hayan migrado.

### app.css completo — **Alto**

Razones:
- 5,652 líneas con 9 componentes activos (actividades, admin, chat, error boundary).
- Las secciones de simulación (wa, call-immersive) suman ~3,861 líneas (~68 % del total).
- No se puede eliminar hasta que **F5 (basicActivities) + F6A-F6F (todas las actividades) + F7 (Admin + ChatDrawer)** estén completas.
- Hay dependencias cruzadas con las clases `app-shell-*` que los shells nuevos también generan.

### Eliminación por sección en app.css (limpieza incremental) — **Medio**

Líneas muertas detectadas (~880 líneas de vistas ya migradas): **Fácil** — se pueden eliminar ahora sin riesgo. Sin embargo, hacerlo en F0 va en contra de la política "no editar código fuente". Candidato a primera acción de F8.

### Secciones de actividades — **Medio a Alto**

- `basicActivities` Concept (F5): ~300 líneas de CSS. Dificultad **Medio** — una sola actividad, clases acotadas.
- `signalActivities` (F6A): ~1,583 líneas (`wa-*`). Dificultad **Alto** — gran volumen, ya semi-migrada a sd-chat-*.
- `CallSimulationActivity` (F6E): ~2,278 líneas (`call-immersive-*`). Dificultad **Alto** — la sección más grande. Múltiples estados (incoming, permissions, active, ended) con diseño sofisticado.
- `InboxActivity` (F6B/C): ~400 líneas. Dificultad **Medio**.
- `ScenarioFlowActivity` (F6D): ~200 líneas. Dificultad **Medio**.
- `WebLabActivity` (F6F): ~300 líneas. Dificultad **Medio**.

### Secciones de admin y chat — **Medio** (F7)

AdminView y ChatDrawer juntos representan ~300 líneas de CSS + 64 reglas `body[data-viewport]`.

### AppErrorBoundary — **Fácil** (puede hacerse en F1)

Solo 8 clases legacy (`app-crash-screen`, `app-crash-card`, `lead`, `eyebrow`). Se puede reemplazar con primitives (SurfaceCard + `sd-eyebrow` + `sd-subtitle`) en < 2 horas. No requiere esperar ninguna fase de actividades.

### SessionLoadingRouteContainer (`page` class) — **Trivial**

1 sola clase (`page`), combinada con `sd-page-shell`. Quitar la palabra `page` del className no tiene riesgo — la misma funcionalidad la da `sd-page-shell`. Fix de 1 línea.

## 9. Hallazgo crítico: signalActivities está a mitad de migración

`signalActivities.jsx` es **única en el set** al mezclar activamente clases legacy con clases nuevas:

- El thread de mensajes usa `sd-chat-*` (clases definidas en `tailwind.css:1175-1622`). ✅ Nuevo.
- El panel lateral (`wa-side-panel`) y los indicadores de señal (`signal-row`, `signal-list`) usan clases de `app.css`. ❌ Legacy.

Esto significa que **la actividad 6A (WhatsApp) fue la única que inició una migración visual parcial** — el thread ya está en el nuevo sistema; el panel lateral no. F6A deberá completar esta migración, no empezarla desde cero.
