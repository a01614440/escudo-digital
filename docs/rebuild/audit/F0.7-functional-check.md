### Análisis del CSS bundle (229.78 kB)

Las tres fuentes CSS que entran al build:
- `frontend/src/styles/legacy.css` → 38,577 B (2,164 líneas)
- `frontend/src/styles/app.css` → 132,213 B (5,652 líneas)
- `frontend/src/styles/tailwind.css` → 44,806 B (1,690 líneas)

Suma de fuentes: **~215 kB raw**. El output CSS es **229.78 kB** — levemente mayor porque Vite procesa y puede añadir prefijos (autoprefixer implícito en Tailwind v4), aunque también elimina comentarios. La diferencia es mínima.

**Desglose estimado post-build:**
- La mayor parte de los 229.78 kB son `app.css` (legacy) + `legacy.css` — juntos representan ~74 % del CSS bundle.
- `tailwind.css` con Tailwind v4 ya genera CSS custom properties + componentes, no utilidades purgadas → su peso no se reduce significativamente por purging en este proyecto.
- **En gzip**: 38.70 kB es razonable pero alto. Una app sin legacy CSS llegaría a ~8-12 kB gzip de CSS útil.

**Implicación:** eliminar `app.css` + `legacy.css` en F8 podría reducir el CSS bundle de **38.70 kB → ~6-8 kB gzip** (~83 % de reducción en transferencia CSS).

### Análisis del JS bundle (460.58 kB)

**Sin code splitting.** El build genera un único chunk JS `index-OWbe3hMB.js` de 460.58 kB raw / 136.55 kB gzip.

Composición estimada:
- React 19 + ReactDOM: ~40 kB gzip
- La app completa (vistas, dominio, actividades, lib): ~90 kB gzip
- `lib/course.js` alone (49.759 kB raw): probablemente ~12-15 kB gzip post-minificación

**136.55 kB gzip para el JS inicial** es el peso total en primera carga — sin lazy loading ni separación de chunks. Para referencia:
- Buena práctica: < 50 kB gzip por chunk inicial
- Tolerable: 50-150 kB
- 136.55 kB: **en el límite superior de "tolerable"**

No hay imports dinámicos (`import(...)`) en ningún archivo del repo → todo el árbol de la aplicación está en el bundle inicial. Las actividades (CallSimulation 27 kB, signalActivities 32 kB, WebLabActivity 22 kB, InboxActivity 15 kB) se cargan aunque el usuario nunca las abra.

## 4. Advertencias que no bloquean pero deben atenderse

### A. Bundle sin code splitting (performance latente)

Severidad: 🔶 media
Descripción: todo el JS en un chunk de 136.55 kB gzip. Las simulaciones (F6) van a añadir peso significativo (CallSimulation + WebLab podrían agregar ~20-30 kB gzip más). Para F9 (rollout) será necesario separar al menos las rutas de actividades.
Acción recomendada: implementar code splitting en `vite.config.js` antes de F9 (no urgente ahora).

### B. CSS bundle dominado por legacy (performance latente)

Severidad: 🔶 media
Descripción: 229.78 kB CSS raw, ~74 % de `app.css` + `legacy.css`. Las reglas legacy incluyen 64 selectores `body[data-viewport]` que son el single greatest consumer.
Acción recomendada: retirar `app.css` + `legacy.css` en F8 (plan existente). Hasta entonces, asumible.

### C. Cero tests para vistas, primitives, patterns y shells

Severidad: 🔶 media-baja (no bloquea desarrollo, sí amplifica riesgo de regresión visual)
Descripción: los 7 archivos de test cubren solo lógica de dominio (course, activities, navigation). No hay tests para ningún componente visual.
Acción recomendada: los smoke tests por bloque que pide `docs/rebuild/04-validation-checklists.md` no se corren automáticamente — por ahora son manuales. Considerar añadir por lo menos test de snapshot/render trivial para primitives críticos en F1.

### D. Tailwind v4 sin purging efectivo de legacy.css

Severidad: 🔶 media-baja
Descripción: Tailwind v4 configurado con `source("../")` en `tailwind.css:3` — escanea todo `frontend/`. Pero `legacy.css` y `app.css` no son Tailwind; se incluyen enteros sin purging. Tailwind v4 no puede purgar CSS que no genera él mismo.
Acción: estructural; se resuelve al eliminar los archivos legacy en F8.

### E. Sin separación dev/prod en variables de entorno

Severidad: 🟡 baja
Descripción: `vite.config.js` no configura `define` para `process.env.NODE_ENV` ni diferencia de vars entre dev y prod. `.env` existe en la raíz pero con valores locales — si alguien hace build en una máquina diferente sin `.env`, el build pasa pero la app en runtime no tiene config de API.
Acción: leve; no afecta la auditoría de F0.
