# F6.C - Category identity / color semantics pass

Fecha: 2026-04-23
Commit base: `08c1976` (`feat(F6.B): harden simulation readability`)

## 1. Objetivo

Cerrar el pase sistemico de identidad por categoria sin entrar todavia al rediseno interno de cada simulacion.

F6.C no intenta terminar WhatsApp, SMS, Inbox, WebLab, CallSimulation o ScenarioFlow por separado. Esta subfase deja una semantica visual comun para que cada familia de simulacion tenga un acento, una superficie, un borde, un glow y atributos estables que puedan usarse en F6.D-F6.I sin repetir hacks locales.

## 2. Cambios realizados

### Metadata compartida de categorias

Archivo: `frontend/src/components/activities/immersive/shared.js`

Se agrego:

- `SIMULATION_CATEGORY_META`
- `getSimulationCategory(activity)`
- `getSimulationCategoryClass(category)`

Mapeo actual:

- `sim_chat` -> `chat`
- `inbox` con `kind="sms"` -> `sms`
- `inbox` sin `kind="sms"` -> `email`
- `web_lab` -> `web`
- `call_sim` -> `call`
- `scenario_flow` -> `scenario`
- `compare_domains` / `signal_hunt` -> `analysis`
- fallback -> `analysis`

Resultado: existe una fuente compartida para identidad de canal, sin tocar `ActivityRenderer.jsx` ni `activityRegistry.js`.

### ActivityChrome

Archivo: `frontend/src/components/activities/sharedActivityUi.jsx`

- `ActivityChrome` resuelve la categoria con `getSimulationCategory(activity)`.
- El chrome guiado y el inmersivo reciben `sd-simulation-category sd-simulation-category-*`.
- Ambos exponen `data-sd-simulation-category`.

Resultado: todas las actividades que pasan por el chrome comun heredan identidad semantica sin cambiar su logica.

### Raices de simulaciones principales

Archivos:

- `frontend/src/components/activities/signalActivities.jsx`
- `frontend/src/components/activities/immersive/InboxActivity.jsx`
- `frontend/src/components/activities/immersive/WebLabActivity.jsx`
- `frontend/src/components/activities/immersive/ScenarioFlowActivity.jsx`
- `frontend/src/components/activities/CallSimulationActivity.jsx`

Cambios:

- WhatsApp/chat declara categoria `chat` y canal `whatsapp`.
- Inbox declara categoria `email` o `sms` segun `kind`.
- WebLab declara categoria `web` y canal `weblab`.
- ScenarioFlow declara categoria `scenario` y canal `scenario-flow`.
- CallSimulation declara categoria `call` y canal `call`.

Resultado: los roots relevantes ahora tienen clases y `data-*` estables para CSS, pruebas y futuros pases F6.x.

### Tailwind layer

Archivo: `frontend/src/styles/tailwind.css`

Se agregaron variables scoped:

- `--sd-simulation-category-accent`
- `--sd-simulation-category-accent-strong`
- `--sd-simulation-category-soft`
- `--sd-simulation-category-surface`
- `--sd-simulation-category-border`
- `--sd-simulation-category-glow`

Categorias definidas:

- `chat`
- `sms`
- `email`
- `web`
- `call`
- `scenario`
- `analysis`

Tambien se conectaron estos tokens a:

- eyebrows
- labels de `ActivitySummaryBar`
- labels de progress pills inmersivos
- badges soft/accent
- summary cards
- readable surfaces
- immersive panels
- aside panels
- progress pills

Se agrego una adaptacion para `body[data-theme='dark']` para evitar que las superficies por categoria queden demasiado claras en tema oscuro.

## 5. Limites respetados

No se tocaron:

- `ActivityRenderer.jsx`
- `activityRegistry.js`
- hooks de dominio
- backend
- services
- contracts
- scoring
- `CoursesView.jsx`
- `LessonView.jsx`
- `app.css`
- `legacy.css`

La subfase toco solo metadata compartida de actividades, roots de simulaciones para clases/atributos semanticos, CSS scoped F6 y tests. No se cambio scoring, flujo, apertura, completion, services ni estructura de datos.
