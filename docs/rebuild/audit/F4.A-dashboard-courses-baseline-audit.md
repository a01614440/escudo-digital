# F4.A - Dashboard / Courses baseline audit

Fecha: 2026-04-23

Commit base validado: `d13bc8c` (`docs(F3.G): close auth survey validation`)

## 1. Resumen breve

F4 puede abrirse ahora porque F1 foundation, F2 shells y F3 Auth + Survey ya quedaron cerradas localmente. La siguiente superficie grande y aun abierta es `CoursesView`, que es el dashboard de ruta y el puente directo hacia `LessonView`.

No conviene saltarse F4 porque aqui viven juntas cuatro piezas que el producto no puede dejar ambiguas:

- generacion y lectura de ruta;
- continuidad y reanudacion;
- progreso y foco;
- ajustes/regeneracion de la cabina.

Si esta pantalla queda a medias, el flujo post-survey se queda sin una lectura clara de "que sigue" y la entrada a lesson sigue arrastrando deuda visual y de composicion.

## 2. Estado actual de CoursesView

`frontend/src/components/CoursesView.jsx` sigue siendo un monolito de 1702 lineas. Aun despues de los cierres F1-F3, esta vista concentra la mayor parte de la experiencia de dashboard y combina:

- composicion visual;

- continuidad de ruta;

Hay cuatro señales claras de deuda de composicion, no de dominio:

1. `RouteNavigatorRail` sigue cargando contraste local fuerte con gradient hardcodeado y overrides de `text-white`:
   - `frontend/src/components/CoursesView.jsx:595-610`

2. Hay overrides locales `!grid-cols-1` sobre `ProgressSummary` en varias ramas:
   - `frontend/src/components/CoursesView.jsx:375`
   - `frontend/src/components/CoursesView.jsx:808`
   - `frontend/src/components/CoursesView.jsx:893`

3. `buildJourneyProgress` se calcula en `CoursesView`, pero `journeySteps` no se renderiza en esta vista:
   - `frontend/src/components/CoursesView.jsx:1379-1387`
   - el componente `JourneyStepper` esta importado en la linea 22, pero no se usa en esta superficie.

4. La rama desktop sigue dependiendo de `WorkspaceLayout`:
   - `frontend/src/components/CoursesView.jsx:1628-1672`

Nada de eso requiere abrir `useCourseProgress`, `courseRules`, `courseService` o `progressService` en F4.A. Son limites de lectura.

## 3. Cobertura y pruebas

### Lo que si esta cubierto

`test/course-dashboard-view-model.test.js` cubre la logica de apoyo del dashboard:

- resume target;
- recommended index y unlocked limit;

`test/course-hardening.test.js` cubre continuidad y routing de estado:

- restore de learning state;
- next activity selection;

`test/foundation-smoke.test.js` ya protege las piezas de foundation que `CoursesView` consume:

- `ProgressSummary`;
- `JourneyStepper`;
- `QuestionPage`;
- `InfoPanel`;
- `AssessmentLayout`;
- `InlineMessage`;
- `SurfaceCard`;
- `Checkbox`;
- `Radio`;
- `IconButton`.

### Lo que aun falta

No existe todavia un test de render o comportamiento visual para `CoursesView` como superficie completa.

No existe una story dedicada para `CoursesView` en `frontend/src/components/*.stories.jsx` o `frontend/src/patterns/*.stories.jsx`.

Eso significa que el dashboard esta protegido por:

- tests de dominio y view-model;
- tests de foundation;
- composicion manual en el monolito.

## 7. Riesgos principales

1. `CoursesView` ya es un monolito grande. Cualquier cambio sin subfase clara va a mezclar demasiadas superficies y hacer dificil la revision.

2. El dashboard sigue arrastrando estilos locales de alto contraste en `RouteNavigatorRail` y overrides de `ProgressSummary`. Eso es deuda visual real, no teoria.

3. `JourneyStepper` y `buildJourneyProgress` ya tienen sentido como concepto, pero aqui no estan integrados. F4 necesita decidir si ese hilo se elimina o se vuelve visible.

4. No hay story dedicada ni render test para `CoursesView`, asi que la validacion de esta pantalla sigue siendo mas fragil que la de foundation.

5. La rama desktop continua dependiendo de `WorkspaceLayout`, lo que hace que F4 deba evitar un redisenio que termine reabriendo shell work.

6. `CoursesView` mezcla estado de presentacion con decisiones de lectura de ruta. Si F4 toca demasiado, corre el riesgo de invadir hooks o services que ya deberian quedar congelados.

## 8. Recomendacion final

La estructura propuesta de F4 si debe mantenerse en subfases pequenas. La version recomendada es:

- F4.B - Dashboard shell / empty states closeout
- F4.C - Route scene hardening
- F4.D - Progress scene hardening
- F4.E - Settings / regeneration hardening
- F4.F - Top-level cleanup and journey decision
- F4.G - F4 closeout validation

Esta division deja `CoursesView` como unico frente de implementacion visible y evita tocar `App`, route containers, hooks de dominio, servicios, shells o backend.

Mi recomendacion concreta es:

- no abrir ninguna pieza de F4 fuera de `CoursesView.jsx` y un test nuevo de dashboard;
- no tocar F1/F2/F3 de nuevo;
- no abrir lesson, admin, chat o simulaciones;
- resolver el hilo de `JourneyStepper` dentro de F4.F, no antes.
