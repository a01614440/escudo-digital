# F5.A - Lesson shell / activity chrome baseline audit

Fecha: 2026-04-23

Commit base validado: `e044b86` (`docs(F4.G): close courses refine validation`)

## Objetivo

Abrir F5 solo como auditoria baseline. No se implementa nada en `LessonView`, actividades, renderer, registry, estilos ni dominio.

F5 existe para cerrar la superficie de practica:

- lesson shell;
- mapa de actividades;
- viewport de actividad;
- chrome generico de actividades;
- feedback y cierre de modulo;
- separacion estricta frente a F6 simulaciones.

F5 no debe convertirse en migracion de WhatsApp, Inbox, CallSimulation, WebLab ni ScenarioFlow. Esas familias pertenecen a F6A-F6F.

## 1. Auditoria breve del estado

F5 ahora si puede abrirse porque F1, F2, F3 y F4 ya dejaron lista la base visual, shells, auth/survey y ruta. `CoursesView` ya entrega el handoff hacia modulo; el siguiente riesgo real esta dentro de la experiencia de practica.

No conviene saltar a F6 porque `LessonView` y el activity chrome todavia definen el marco que recibira cualquier simulacion. Si ese marco sigue duplicando instrucciones, comprimiendo el viewport o mezclando responsabilidades, cada simulacion arreglara sus problemas localmente.

La auditoria baseline es necesaria porque `LessonView.jsx` ya parece new-ui por fuera, pero las actividades renderizadas todavia combinan foundation, chrome local y legacy CSS. F5 debe separar lo que corresponde al shell de lesson de lo que pertenece a cada simulacion.

## 2. Estado actual de LessonView

### Que esta bien

- `LessonView.jsx` ya usa foundation y patterns: `SplitHeroLayout`, `WorkspaceLayout`, `StageHero`, `SupportRail`, `ProgressSummary`, `StatStrip`, `ActionCluster`, `KeyValueBlock`, `PanelHeader`, `SurfaceCard`, `Badge`, `Button` e `InlineMessage`.
- La vista ya separa regiones claras: hero de modulo, rail de mando, stage de actividad, rail de insight y cierre de modulo.
- El shell responsive ya distingue mobile, tablet y desktop con `getShellFamily(viewport)`.
- Mobile mantiene stack simple; tablet usa dos zonas; desktop usa `WorkspaceLayout`.
- `ModuleComplete` existe como cierre dedicado y no depende de `ActivityRenderer`.
- El mapa de actividades ya expone estado actual/completado/pendiente con `aria-current="step"` cuando aplica.

### Que esta mal o dudoso

- `LessonView.jsx` tiene 776 lineas y sigue mezclando derivacion, layout responsive, rails, hero, actividad, insight y cierre.
- `LessonActivityStage` renderiza instrucciones ("Que debes hacer" / "Como se evalua") y luego `ActivityRenderer` vuelve a envolver actividades con `ActivityChrome`, que tambien renderiza instrucciones y guia. Hay duplicacion conceptual.
- El viewport real de actividad esta dentro de varias superficies anidadas: `LessonActivityStage` -> `SurfaceCard spotlight` -> instruction cards -> `SurfaceCard panel` -> `ActivityRenderer` -> `ActivityChrome`.
- Desktop mantiene tres columnas para lesson. La idea es valida, pero F5 debe verificar que la practica no quede comprimida por rails laterales.
- `LessonInsightRail` repite progreso/contexto y puede competir con la actividad principal.
- El cierre de modulo esta separado, pero requiere validacion de jerarquia, feedback y CTA para no sentirse como otra pantalla informativa.

### Deuda viva

- No hay test dedicado de F5 para guardar el contrato de `LessonView`, `ActivityRenderer`, `ActivityChrome` y actividades basicas.
- `LessonView` ya esta en new-ui por fuera, pero actividades basicas siguen usando clases legacy (`concept-*`, `activity-actions`, `btn`, `option-grid`, `result-card`).
- `signalActivities.jsx` esta en estado hibrido: seccion nueva `sd-chat-*` y rama antigua `wa-*` conviven. Eso es importante, pero pertenece mayormente a F6A, no a F5.
- `app.css` y `legacy.css` todavia sostienen muchas clases de actividades. El cleanup masivo no pertenece a F5; F5 solo debe reducir dependencia donde toca la practica basica.

## 3. Estado actual de ActivityRenderer y registry

### ActivityRenderer

`frontend/src/components/activities/ActivityRenderer.jsx` es un dispatcher pequeno:

- llama `useSimulationEngine({ moduleId, activityId })`;
- sanitiza `module` y `activity` con `sanitizeScenarioContent`;
- calcula `compact` desde `viewport`;
- resuelve el componente por tipo con `resolveActivityComponent`;
- envuelve todo en `ActivityChrome`;
- pasa `answers`, `assessment`, `startedAtRef` y `onComplete`.

Este archivo es contrato sensible. F5 no debe reescribirlo. Si F5 necesita cambiar el chrome generico, la primera opcion debe ser tocar `sharedActivityUi.jsx` y/o el contenedor `LessonView`, no cambiar el dispatcher ni la forma del registry.

### activityRegistry

`frontend/src/components/activities/activityRegistry.js` registra:

- `concepto`
- `quiz`
- `simulacion`
- `checklist`
- `abierta`
- `sim_chat`
- `compare_domains`
- `signal_hunt`
- `inbox`
- `web_lab`
- `call_sim`
- `scenario_flow`

El fallback es `ConceptActivity`. Este archivo es contrato de dominio/presentacion sensible y debe permanecer congelado salvo autorizacion humana explicita.

## 4. Estado actual de ActivityChrome / sharedActivityUi

`frontend/src/components/activities/sharedActivityUi.jsx` contiene:

- `Paragraphs`
- `SimulationGuide`
- `ActivityChrome`
- `completeActivity`
- `buildActivityFeedback`
- `formatPercent`
- `ActivitySummaryBar`

`ActivityChrome` tiene dos comportamientos:

- para `sim_chat`, devuelve solo `sd-chat-activity-shell` y evita el chrome generico;
- para el resto, renderiza header, descripcion, instrucciones, `SimulationGuide` y children.

El problema no es que exista `ActivityChrome`; el problema es que hoy no esta claro si la instruccion principal vive en `LessonActivityStage` o en `ActivityChrome`. F5 debe cerrar esa frontera.

## 5. Estado actual de feedback

`frontend/src/components/FeedbackPanel.jsx` ya esta bastante alineado con foundation:

- usa `SurfaceCard`;
- usa `PanelHeader`;
- usa `InlineMessage`;
- usa `Badge`;
- maneja feedback string/objeto;
- expone estados de fortaleza, oportunidad, detalle y score.

F5 debe conservarlo y endurecerlo si hace falta, no reescribirlo desde cero. El foco debe ser que actividades basicas lo usen de forma consistente y accesible.

## 12. Riesgos principales

### Visuales

- Convertir F5 en redisenio de simulaciones y perder el foco del shell.
- Hacer el stage mas bonito pero seguir dejando el viewport comprimido.
- Repetir progreso, instrucciones y ayudas hasta que la actividad deje de mandar.

### Funcionales

- Romper `onActivityComplete` al mover wrappers.
- Tocar `ActivityRenderer` y cambiar props esperadas por actividades.
- Romper el fallback del registry o tipos existentes.
- Cambiar payloads de `completeActivity`/feedback sin querer.

### A11y

- Duplicar headings/instrucciones y crear ruido para lector de pantalla.
- Mover feedback sin `role`/`aria-live` apropiado.
- Dejar controles de quiz/checklist/open answer con labels pobres o foco ambiguo.

### Scope creep

- Intentar arreglar WhatsApp en F5.
- Intentar arreglar WebLab/CallSimulation porque se ven mal dentro de lesson.
- Limpiar `legacy.css` completo antes de F8.
