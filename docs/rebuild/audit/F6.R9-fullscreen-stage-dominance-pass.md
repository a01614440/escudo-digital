# F6.R9 - Fullscreen stage dominance pass

Fecha: 2026-04-24
Estado: cerrada localmente

## Objetivo

Hacer que las simulaciones ganen presencia real de pantalla despues de F6.R8, sin reabrir dominio, scoring, backend, renderer ni registry.

F6.D ya habia dado un primer baseline fullscreen. F6.R9 endurece esa base para que el stage sea mas dominante, los rails queden subordinados y las practicas se sientan menos encerradas.

## Cambios implementados

### 1. Stage inmersivo con mas alto y ancho util

Archivo: `frontend/src/styles/tailwind.css`

Se elevaron los limites compartidos del stage:

- `--sd-simulation-stage-min-block: clamp(42rem, 86vh, 82rem)`
- `--sd-simulation-stage-max-inline: 104rem`
- `sd-simulation-main-stage` sube a `min-height: clamp(34rem, 70vh, 68rem)`
- mobile sube a `clamp(36rem, 82vh, 64rem)`

Resultado:

- las simulaciones tienen mas presencia vertical;
- el stage puede ocupar mas ancho real;
- se reduce la sensacion de practica comprimida dentro del lesson.

### 2. Hook explicito de fullscreen focus

Archivos:

- `frontend/src/components/activities/sharedActivityUi.jsx`
- `frontend/src/components/activities/signalActivities.jsx`
- `frontend/src/components/activities/immersive/InboxActivity.jsx`
- `frontend/src/components/activities/immersive/WebLabActivity.jsx`
- `frontend/src/components/activities/immersive/ScenarioFlowActivity.jsx`
- `frontend/src/components/activities/CallSimulationActivity.jsx`

Se agrego `data-sd-stage-focus="fullscreen"` en:

- `ActivityChrome` inmersivo;
- WhatsApp/chat;
- Inbox/SMS/email main stage;
- WebLab workbench;
- ScenarioFlow main stage;
- CallSimulation root.

Resultado:

- el layout declara de forma estable que la practica es la superficie principal;
- los tests pueden proteger que futuras fases no vuelvan a achicar el stage por accidente.

### 3. Rails subordinados al contenido principal

Archivos:

- `frontend/src/components/activities/signalActivities.jsx`
- `frontend/src/components/activities/immersive/InboxActivity.jsx`
- `frontend/src/components/activities/immersive/WebLabActivity.jsx`
- `frontend/src/components/activities/immersive/ScenarioFlowActivity.jsx`
- `frontend/src/styles/tailwind.css`

Se agrego `data-sd-stage-rail="subordinate"` a:

- insight lateral de WhatsApp/chat;
- sidebar de Inbox/SMS/email;
- hint lateral de WebLab;
- aside del workbench de WebLab;
- aside de ScenarioFlow.

La CSS ahora hace que esos rails:

- se mantengan `sticky` solo en desktop;
- tengan `max-height: min(100%, 74vh)`;
- hagan overflow interno si crecen;
- no compitan con el ancho principal.

Resultado:

- los secundarios siguen disponibles;
- la actividad deja de perder protagonismo por paneles laterales permanentes.

### 4. Ratios desktop mas content-first

Archivo: `frontend/src/styles/tailwind.css`

Se reajustaron los ratios desktop:

- Inbox/SMS/email: `minmax(14rem, 18rem) minmax(0, 1fr)`
- WebLab: `minmax(0, 4.2fr) minmax(14rem, 0.72fr)`
- ScenarioFlow: `minmax(0, 3.2fr) minmax(14rem, 0.62fr)`
- Briefing WebLab: `minmax(0, 2.25fr) minmax(14rem, 0.62fr)`

Resultado:

- los rails se estrechan;
- WebLab y ScenarioFlow dan mas ancho al contenido interactivo;
- Inbox conserva lista/detalle sin volver a tres columnas.

### 5. Chat y llamadas con mas espacio de juego

Archivo: `frontend/src/styles/tailwind.css`

WhatsApp/chat:

- `sd-chat-sim` sube de `76rem` a `94rem`;
- desktop pasa a `minmax(0, 2.25fr) minmax(16rem, 0.65fr)`;
- thread sube a `min-height: clamp(30rem, 62vh, 50rem)`;
- thread sube a `max-height: clamp(38rem, 76vh, 64rem)`;
- surface tablet/desktop sube a `min-height: clamp(42rem, 80vh, 72rem)`.

CallSimulation:

- shell sube a `width: min(100%, 92rem)`;
- phone sube a `min-height: clamp(40rem, 78vh, 72rem)`;
- transcript sube a `min-height: clamp(24rem, 54vh, 44rem)`.

Resultado:

- chat se siente mas conversacional y menos angosto;
- llamadas ganan presencia de pantalla;
- ambos quedan mejor preparados para presentacion/demo.

### 6. Guards de regresion

Archivos:

- `test/f6-fullscreen-stage-dominance-pass-smoke.test.js`
- `test/f6-stage-dominance-smoke.test.js`
- `test/f5-comfort-responsive-smoke.test.js`
- `test/f6-whatsapp-chat-smoke.test.js`

Se agrego un guard nuevo para F6.R9 y se actualizaron guards heredados para defender los nuevos limites.

Protege:

- `data-sd-stage-focus="fullscreen"`;
- rails subordinados;
- variables `--sd-simulation-stage-min-block` y `--sd-simulation-stage-max-inline`;
- ratios desktop content-first;
- dimensiones expandidas de chat y calls.
