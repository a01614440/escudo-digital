# F5.E - Lesson shell layout refactor (content-first / fullscreen)

Fase: F5.E.
Alcance: hacer que `LessonView` deje de forzar workspace de tres columnas, que la actividad principal mande, y que las actividades inmersivas entren en un contenedor fullscreen/content-first sin chrome generico.

## Objetivo

Resolver los problemas #11, #13, #14, #15, #17 parcial y #18 del baseline F5.A:

- `LessonView` forzaba `WorkspaceLayout` 3-col para cualquier actividad;
- actividades inmersivas seguian comprimidas entre rail izquierdo, main y rail derecho;
- `ActivityChrome` solo liberaba `sim_chat`, dejando Inbox/WebLab/Call/Scenario con chrome clasico;
- el insight rail aparecia aunque no hubiera suficiente historial;
- el renderer tenia demasiados wrappers antes de llegar a la practica.

Regla central aplicada: **content-first, not dashboard-first**.

## Implementacion

### `frontend/src/components/LessonView.jsx`

- Se agrego deteccion pasiva de actividades inmersivas con `IMMERSIVE_ACTIVITY_TYPES`:
  - `sim_chat`
  - `inbox`
  - `web_lab`
  - `call_sim`
  - `scenario_flow`
- Se agrego `getLessonStageMode(activity)` para resolver:
  - `immersive`
  - `guided`
- Se elimino el uso de `WorkspaceLayout` dentro del render principal de `LessonView`.
- El root de lesson ahora expone `data-sd-activity-mode={stageMode}`.
- El layout principal ahora usa:
  - `data-sd-lesson-layout="immersive-fullscreen"` para inmersivas;
  - `data-sd-lesson-layout="guided-two-pane"` para actividades guiadas/basicas.
- `LessonActivityStage` ahora tiene dos ramas:
  - `immersive`: renderiza el `ActivityRenderer` directamente dentro de `sd-lesson-stage-immersive`, sin `SurfaceCard spotlight`, sin wrapper panel interno y sin briefing visible duplicado;
  - `guided`: mantiene el stage guiado, pero el renderer ya no vive dentro de otra `SurfaceCard`; ahora usa `sd-lesson-renderer-frame`.
- `LessonCommandRail` acepta `compact`.
  - En modo inmersivo se renderiza como una card compacta debajo de la practica, con el mapa en `<details>`.
  - En modo guiado conserva el rail normal.
- `LessonInsightRail` ya no aparece cuando no hay historial ni ultima reentrada.
- El layout inmersivo pone primero la practica y mueve contexto/mapa/insight a una region secundaria debajo.
- El layout guiado queda en dos paneles: contexto + actividad, no tres columnas rigidas.

### `frontend/src/components/activities/sharedActivityUi.jsx`

- `ActivityChrome` ahora reconoce el mismo set de inmersivas:
  - `sim_chat`
  - `inbox`
  - `web_lab`
  - `call_sim`
  - `scenario_flow`
- Para esas actividades devuelve solo un shell `sd-immersive-activity-shell`.
- `sim_chat` conserva tambien `sd-chat-activity-shell` para no romper estilos existentes.
- Se agregaron `data-sd-activity-chrome="immersive"` y `data-sd-activity-type`.
- No se cambio `ActivityRenderer` ni `activityRegistry`.

### `frontend/src/styles/tailwind.css`

- Se agregaron hooks de layout:
  - `.sd-lesson-layout`
  - `.sd-lesson-layout-guided`
  - `.sd-lesson-layout-immersive`
  - `.sd-lesson-stage`
  - `.sd-lesson-stage-immersive`
  - `.sd-immersive-activity-shell`
- El modo inmersivo usa una sola columna para la practica.
- La region secundaria inmersiva se vuelve grid responsive debajo del stage.
- Se aseguro `min-width: 0` en stage, main, context, renderer frame y shell inmersivo para evitar overflow/compresion rara.

### Tests

Se agrego `test/f5-lesson-shell-smoke.test.js` con 4 guards:

1. `LessonView` elige layout dinamico y ya no renderiza `WorkspaceLayout`.
2. `LessonActivityStage` entrega actividades inmersivas al renderer sin cards de lesson anidadas.
3. `ActivityChrome` bypasses el chrome generico para todas las inmersivas.
4. Tailwind contiene los hooks de fullscreen/content-first.

## Hallazgos

### Lo que si cambia

- La practica inmersiva deja de vivir dentro del workspace 3-col.
- `sim_chat`, `inbox`, `web_lab`, `call_sim` y `scenario_flow` ya no reciben el chrome clasico de `ActivityChrome`.
- El contexto del modulo deja de robar ancho principal en inmersivas.
- El mapa del modulo en inmersivas queda disponible pero secundario/collapsable.
- El insight rail desaparece cuando no hay historial que justifique su presencia.

### Lo que no cambia

- No se toco `ActivityRenderer.jsx`.
- No se toco `activityRegistry.js`.
- No se tocaron simulaciones especificas.
- No se tocaron hooks de dominio.
- No se tocaron services, backend, contracts ni scoring.
- No se toco `legacy.css` ni `app.css`.
- No se hizo el pase F5.F de instructions/feedback.

## Riesgos residuales

- `LessonActivityStage` guiado todavia mantiene briefing visible y `ActivityChrome` tambien puede mostrar instrucciones. Eso es F5.F.
- `ModuleComplete` sigue con estructura tipo split/rail. Se debe compactar en F5.F o F5.G segun prioridad.
- El insight rail se redujo por condicion de historial, pero su contenido interno todavia conserva lectura completa cuando aparece. F5.G puede compactarlo mas.
- El set de inmersivas es declarativo por tipo; si aparece un nuevo tipo inmersivo futuro, habra que agregarlo en `LessonView` y `sharedActivityUi`.
