# F2.D - Shell contract + navigation hardening

Fecha: 2026-04-23

## Objetivo

Reforzar el contrato de shells, slots y route-containers sin reconstruir la arquitectura ni tocar vistas grandes.

Esta subfase cierra la parte tecnica de F2.D: hacer que los shells expongan de forma estable la politica declarativa que ya usan internamente, proteger la semantica de landmarks y corregir el caso de loading que todavia traia wrapper legacy.

## Cambios realizados

### 1. Shell policy expuesta como contrato estable

Archivos:

- `frontend/src/shells/MobileShell.jsx`
- `frontend/src/shells/TabletShell.jsx`
- `frontend/src/shells/DesktopShell.jsx`

Se agregaron data attributes derivados de `shellPolicies.js`:

- `data-shell-header-mode`
- `data-shell-secondary-persistent`
- `data-shell-floating-mode`
- `data-shell-slot-order`

Los shells ya exponian:

- `data-shell-family`
- `data-shell-intent`
- `data-shell-primary-mode`
- `data-shell-secondary-mode`
- `data-shell-overlay-mode`

Con esto, el contrato real de shell queda visible y testeable desde la capa DOM sin acoplar vistas a implementaciones internas.

### 2. Semantica de slots protegida

Archivo:

- `test/shell-architecture.test.js`

Se agrego cobertura source-level para confirmar que `MobileShell`, `TabletShell` y `DesktopShell` mantienen:

- un slot primario renderizado como `<main data-shell-slot="primary">`;
- slot de header estable;
- slot secundario estable;
- slot floating estable;
- slot overlay estable.

Tambien se verifica la diferencia intencional por familia:

- mobile usa region secundaria en `div`;
- tablet y desktop usan `aside` para region secundaria.

### 3. Loading route sin nested main ni wrapper legacy

Archivo:

- `frontend/src/route-containers/SessionLoadingRouteContainer.jsx`

Se cambio el wrapper interno de:

```jsx
<main className="page sd-page-shell">
```

a:

```jsx
<div className="sd-page-shell">
```

Motivo:

- `RouteContainer` ya vive dentro del `<main data-shell-slot="primary">` del shell.
- Mantener otro `<main>` adentro generaba nested landmark.
- La clase `page` era arrastre legacy innecesario para este route-container.

## Tests agregados

Archivo:

- `test/shell-architecture.test.js`

Cobertura nueva:

- los tres shells exponen el contrato de policy como data attributes;
- los slots principales conservan semantica estable;
- `SessionLoadingRouteContainer` no introduce `<main>` anidado;
- `SessionLoadingRouteContainer` no vuelve a usar `className="page..."`.
