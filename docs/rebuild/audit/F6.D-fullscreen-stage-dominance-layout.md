# F6.D - Fullscreen / stage-dominance / layout pass

Fecha: 2026-04-23
Commit base: `b65c843` (`feat(F6.C): add simulation category semantics`)

## 1. Objetivo

Cerrar el pase compartido de dominancia fullscreen para simulaciones: que el stage principal tenga mas ancho, mas altura util y menos sensacion de estar encerrado dentro de un dashboard.

F6.D no redisena todavia WhatsApp, SMS, Inbox, WebLab, CallSimulation o ScenarioFlow por dentro. Esta subfase deja hooks y reglas de layout para que F6.E-F6.I puedan refinar cada canal encima de una base con mas espacio real.

## 2. Cambios realizados

### ActivityChrome inmersivo

Archivo: `frontend/src/components/activities/sharedActivityUi.jsx`

- El chrome inmersivo ahora declara `data-sd-stage-dominance="primary"`.
- Tambien declara `data-sd-stage-layout="fullscreen"`.

Resultado: el wrapper comun ya comunica que la simulacion debe ser la superficie principal, sin tocar `ActivityRenderer.jsx` ni `activityRegistry.js`.

### Raices de simulaciones

Archivos:

- `frontend/src/components/activities/signalActivities.jsx`
- `frontend/src/components/activities/immersive/InboxActivity.jsx`
- `frontend/src/components/activities/immersive/WebLabActivity.jsx`
- `frontend/src/components/activities/immersive/ScenarioFlowActivity.jsx`
- `frontend/src/components/activities/CallSimulationActivity.jsx`

Cambios:

- WhatsApp/chat, SMS/Inbox, WebLab, ScenarioFlow y CallSimulation declaran `data-sd-stage-dominance="primary"`.
- Inbox declara `sd-simulation-main-stage` con `data-sd-stage-layout="list-detail"`.
- WebLab separa briefing (`sd-simulation-briefing-strip`) de stage principal (`sd-simulation-main-stage`) y marca `data-sd-stage-layout="weblab-workbench"`.
- ScenarioFlow declara `sd-simulation-main-stage` con `data-sd-stage-layout="scenario-flow"`.

Resultado: cada familia inmersiva queda preparada para reglas de fullscreen sin mezclar scoring, estado ni datos.

### Tailwind layer

Archivo: `frontend/src/styles/tailwind.css`

Se reforzo:

- `.sd-lesson-layout-immersive`
- `.sd-lesson-stage-immersive`
- `.sd-immersive-activity-shell`
- `.sd-simulation-category[data-sd-stage-dominance='primary']`
- `.sd-simulation-main-stage`
- `.sd-simulation-briefing-strip`

Cambios de layout:

- El stage inmersivo sube a `--sd-simulation-stage-min-block: clamp(38rem, 80vh, 72rem)`.
- Mobile usa fallback propio `clamp(34rem, 74vh, 60rem)`.
- El stage y primary container pueden llegar a `96rem`.
- Los summaries se centran y conservan ancho estable sin bloquear el stage.
- `list-detail`, `weblab-workbench` y `scenario-flow` tienen proporciones especificas de mayor dominancia para el area principal.
- CallSimulation recibe overrides scoped para mantener el telefono/transcript como pieza dominante sin tocar `app.css`.

### WhatsApp / chat stage

Archivo: `frontend/src/styles/tailwind.css`

- `sd-chat-sim` deja el limite estrecho `46rem` y pasa a `76rem`.
- El thread pasa de altura fija `22rem` / `32rem` a clamps mas amplios:
  - `min-height: clamp(26rem, 54vh, 42rem)`
  - `max-height: clamp(34rem, 68vh, 54rem)`
- En tablet/desktop, `sd-chat-surface` gana altura minima fullscreen.

Resultado: chat ya no queda encerrado en una columna estrecha antes de empezar F6.E.

## 5. Limites respetados

No se tocaron:

- `ActivityRenderer.jsx`
- `activityRegistry.js`
- hooks de dominio
- backend
- services
- contracts
- scoring
- `CoursesView.jsx`
- `LessonView.jsx`
- `app.css`
- `legacy.css`

La subfase solo agrego atributos de stage/layout, CSS scoped en `tailwind.css`, tests y documentacion. No cambio flujo, apertura, completion, llamadas a IA/audio ni estructura de datos.

## 6. Deuda que sigue viva

Queda fuera de F6.D:

- refinamiento interno de WhatsApp/chat;
- separacion visual real SMS vs Inbox;
- WebLab como pagina clonada mas creible;
- CallSimulation con menos informacion simultanea;
- ScenarioFlow como experiencia narrativa;
- feedback/completion mas integrado por familia;
- cleanup scoped de `app.css` / `legacy.css` cuando la fase lo permita.
