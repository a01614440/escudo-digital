## 3. SurveyView.jsx — 961 líneas

| Dimensión | Veredicto |
|---|---|
| Tamaño | 🔶 961 líneas — alto, pero decomposición en 21 sub-componentes lo hace navegable |
| Usa shells | 🔶 parcial (igual que AuthView: sub-componentes ramifican por `shellFamily`) |

| Usa patterns/layouts | ✅ 2 layouts + 9 patterns = **uso máximo de foundation** |
| Legacy deps | 🟢 **0 clases legacy** |
| Complejidad | 🔶 21 funciones internas, 2 `useState`, 3 `useEffect`, 2 `useMemo` — alto volumen, arquitectura correcta |
| Veredicto | 🔶 refinar |

**Arquitectura interna (stage machine):**

La vista implementa un mini state-machine visual por `surveyStage` (`survey` | `loading` | `results`) y `showIntro`:

```
SurveyView
├── IntroScene            (showIntro === true)
│   ├── SurveyStageHero   (tone editorial)
│   ├── IntroActionPanel  (panel de CTA + metadatos)
│   └── IntroSupportBand  (rail editorial)
├── SurveyStageScene      (surveyStage === 'survey' && !showIntro)
│   ├── SurveyStageHero   (tone spotlight)
│   └── WorkspaceLayout 3 col
│       ├── SurveyCommandDeck  (progress + journey)
│       ├── QuestionBoard      (SurveyChoiceCard + renderInput)
│       └── SurveyInsightDeck  (hints + formato)
├── LoadingScene          (surveyStage === 'loading')
│   ├── SurveyStageHero   (tone editorial)
│   └── SplitHero
│       ├── LoadingPrimaryPanel
│       └── LoadingSupportBand
└── ResultsScene          (surveyStage === 'results')
    ├── SurveyStageHero   (tone spotlight, aside con nivel)
    └── WorkspaceLayout 3 col
        ├── ResultsCommandDeck  (profile summary)
        ├── ResultsMainDeck     (result body)
        └── ResultsRouteRail    (CTA + course error + restart)
```

**Puntos fuertes:**
- Cada escena es una composición de hero + layout + 2–3 rails — patrón reutilizable.
- `getStageHeroModel` centraliza la copia/metadatos del hero por stage (línea 42-156). Buena abstracción.
- `SurveyChoiceCard` (219-240) encapsula la interacción radio/checkbox envolviendo `SurfaceCard interactive` + `<input class="sr-only">`.
- `renderInput` (242-310) maneja 4 tipos de pregunta (`single`/`multi`/`select`/texto libre) con helpers uniformes.

**Hallazgos mayores:**
- **Choice cards reinventan Checkbox/Radio**: `SurveyChoiceCard` en línea 219 usa `<input type="radio|checkbox" class="sr-only">` + `<SurfaceCard>` como label. Es exactamente el pattern que un **`Checkbox` / `Radio`** primitive cubriría. Sin esos primitives (F0.3 los marcó como faltantes), cada vista se arma el suyo con riesgos: `<SurfaceCard interactive>` **no tiene keyboard support** (F0.3) — aquí funciona porque el `<input>` nativo sí lo tiene y el SurfaceCard es label, no el target focuseable. OK por accidente, no por diseño.
- **Línea 228**: `shadow-[0_24px_50px_-28px_rgba(47,99,255,0.3)]` — shadow hardcoded en choice card seleccionada.
- **Línea 610**: `shadow-[0_30px_80px_-42px_rgba(16,33,61,0.9)]` — mismo hack que AuthView (panel command con texto blanco forzado). Confirma que este patrón se repite y merece ser primitive.
- **Línea 611**: `[&_.text-sd-text]:text-white [&_.text-sd-muted]:text-white/76` — el mismo override de tokens que AuthView.

**Deudas estructurales:**
- **Línea 559-593** (`SurveyStageScene`): crea su propio `WorkspaceLayout` con `className` de grid custom (`xl:grid-cols-[16.5rem_1.28fr_18.5rem]`) — el patrón "override grid por vista" se repite. Esto **acopla la vista a un breakpoint-específico tweak**; si en F2 se ajustan los shells, hay que volver a tocar aquí.
- **Línea 921**: `<section id="surveyView" className="sd-page-shell py-[var(--sd-shell-padding-block)]">` — doble wrap igual que AuthView.
- **Línea 888-907**: `useState(showIntro)` inicializado desde props + `useEffect` para sincronizar con cambios de `surveyStage`/`surveyIndex`/`hasAnswers` → anti-pattern clásico "derive state from props". Mejor: computarlo como `useMemo` o elevarlo al hook.
- Profundidad JSX en `ResultsScene` (líneas 806-858): 6 niveles de anidamiento. El `<StageHero>` ya aporta 3-4 niveles internos.

**Micro-hallazgos:**
- Línea 450: `details className="rounded-[24px] border border-sd-border bg-white/88"` — `rounded-[24px]` arbitrary (no escala tokens).
- Línea 191: `bg-white/84` — literal white.
- Línea 189-195: aside de "Nivel estimado" usa `font-display text-5xl leading-[0.92] tracking-[-0.06em]` — tipografía inline sin token (confirma hallazgo typography en F0.2).

**Conclusión:** SurveyView es correcta y funciona; el tamaño es consecuencia de la ausencia de `QuestionPage` y `AssessmentLayout`. **Refinar**, no rehacer: factorizar `SurveyChoiceCard` → `Radio`/`Checkbox` primitives, cristalizar el patrón de stage machine en un layout genérico.

- `getModuleCtaLabel` (línea 97-102) es una función local sin memo — recreada cada render.

**Conclusión:** CoursesView es **funcionalmente correcta** pero arquitectónicamente ha llegado al límite. El tamaño no viene de código sucio, sino de **ausencia de `TaskList`, `ListDetail`, `CourseLayout`** (F0.5). Rehacer el shell visual (no la lógica) dentro de F4 — 1,700 líneas es fácilmente reducible a ~700 si se introducen esos 3 patterns.

## 11. Patrones recurrentes descubiertos

Estos patrones **aparecen 3 o más veces** entre las vistas migradas y son candidatos a foundation:

1. **Override de tono inverso en SurfaceCard variant command/hero** — aparece en AuthView:161, SurveyView:610, CoursesView:609. Hack: `[&_.text-sd-text]:text-white [&_.text-sd-muted]:text-white/76 [&_.sd-eyebrow]:text-white/70 [&_.sd-heading-sm]:text-white [&_.sd-copy-sm]:text-white/76`. **Debe ser prop `tone="inverse"` en SurfaceCard**.
2. **3-branch por shellFamily con replicación de children** — aparece en CoursesView (tab ruta), LessonView (render final). Los 3 branches duplican 3 sub-componentes con wrappers distintos. **Debe ser layout de dominio (`CourseLayout`, `LessonLayout`) + container queries, no shellFamily-switch**.
3. **Wrapper `<section className="sd-page-shell py-[...]" data-sd-container="true">` al inicio de cada vista** — AuthView:230, SurveyView:922, CoursesView:152+1490, LessonView:92+529+744. **El shell externo (Mobile/Tablet/Desktop) ya envuelve `sd-page-shell`**; este wrap es redundante. Ya reportado en F0.4 §8.2.
4. **`SplitHeroLayout className={shellFamily === 'tablet' ? '...' : shellFamily === 'desktop' ? '...' : ''}`** — 4 vistas lo hacen (AuthView:234-240, SurveyView:366-373, CoursesView:155-160+1497-1501, LessonView via WorkspaceLayout). El layout ya recibe `shellFamily`; duplicar con `className` es code smell. **El layout debería absorber la variación**.
5. **Choice cards con `<input class="sr-only">`** — SurveyView:222 (único sitio hoy). Si se añade un second caller (quiz en actividades), ya no será único. **Checkbox/Radio primitives cubren este caso**.

## 12. Riesgos para fases siguientes

1. **F3 (Auth+Survey)** — AuthView ya está bien; SurveyView necesita Checkbox/Radio primitives + factorización en `AssessmentLayout`. Si F3 se abre sin esos, las 961 líneas de SurveyView van a seguir creciendo con cada ajuste.
2. **F4 (Dashboard)** — CoursesView es **el mayor cuello de botella del rebuild**. 1,702 líneas con 3-branch repetido. Si F4 se abre sin primero construir `TaskList`/`ListDetail`/`CourseLayout` (F0.5), cualquier cambio en CoursesView multiplica la deuda.
3. **F5 (Lesson)** — LessonView es viable de refinar dentro de F5 con un `LessonLayout`; la dependencia `sd-lesson-mission-hero` con !important debe limpiarse como parte del mismo fix.
4. **Patrón "override inverso"** — se usa 3 veces. Si no se formaliza como prop de `SurfaceCard`, cualquier vista que añada variant hero/command en F6 (simulaciones) repetirá el hack `[&_.text-sd-text]:text-white`. Esto **afecta a todas las simulaciones inmersivas**.
5. **`InlineMessage` sin role/aria-live** (F0.3) se usa en FeedbackPanel con tono variable. F5 expone este feedback al flujo de actividad — **bloqueo a11y real** al completar actividad con score bajo.
