### 3.1 Observación transversal a los layouts

Ambos layouts **reciben `shellFamily` como prop** y lo usan para elegir el grid template. Esto es **redundante** con lo que ya hace `DeviceShell`: el shell padre ya organiza macro. Si un layout se usa dentro de un shell concreto, el shell debería imponer su macro y el layout solo definir slots semánticos.

La consecuencia práctica es que al cambiar de shell, tanto el shell como el layout re-computan columnas — dos fuentes de verdad para el mismo concepto. Y si una vista quiere un SplitHeroLayout en desktop con grid distinto al default del shell, tiene que pasar props custom.

Un diseño más limpio sería:
- Layout define SLOTS semánticos, sin decidir columns.
- Las classes CSS del layout usan container queries (`data-sd-container` + `@container`) para adaptarse al espacio útil real, no al shell global.

La implementación actual ya marca `data-sd-container="true"` en el `<section>` raíz (SplitHeroLayout:30, WorkspaceLayout:30) — está preparada para container queries. Pero **no hay `@container` queries escritas para los layouts**; en vez de eso usa `md:grid-cols-[...]` (media queries).

**Veredicto layouts: 🔶 — funcionales y semánticamente correctos, pero mezclan responsabilidades de shell-family.**

## 8. Micro-hallazgos

1. **`JourneyStepper.jsx:24`** — shadow hardcoded rgba(47,99,255,0.34). Única grieta de tokens en patterns/layouts.
2. **`JourneyStepper` usa clases Tailwind inline** (`border-sd-accent bg-sd-accent-soft`) en lugar de una clase `sd-journey-stepper-*` en tailwind.css — inconsistencia con el patrón "clases sd-* compiladas" del resto de patterns.
3. **`SectionHeader.jsx:39`** — hardcodea `<h2>` siempre. No hay prop `as` para controlar el nivel de heading. En patterns que se anidan dentro de otros (ej. `PanelHeader` dentro de un `StageHero`), el orden semántico de headings puede quedar desordenado (`<h2>` del StageHero con `<h2>` del PanelHeader en el mismo nivel).
4. **`PanelHeader.jsx:15`** — hardcodea `<h3>`. Relacionado con el punto anterior.
5. **`SplitHeroLayout` y `WorkspaceLayout` reciben `shellFamily` como prop** — doble acoplamiento con el shell padre (ver §3.1).
6. **`FoundationPatterns.stories.jsx:97`** — texto dice "11 Primitives" pero eran 15 en F0.3. Deuda de documentación.
7. **`FoundationLayouts.stories.jsx` no muestra el comportamiento responsive** — los demos fijan `shellFamily="desktop"`. No hay controles para probar mobile/tablet en Storybook.
8. **Ningún pattern o layout tiene test** en `test/` — cero cobertura unitaria (el test `foundation-smoke` solo verifica que los tokens tengan ciertas claves; no renderiza patterns).
9. **`MetricCard.jsx:54`** — `hint` se renderiza **dentro del contenedor `<SurfaceCard>`** pero **fuera del grid `<div className="grid gap-2">`**. Si se usa `hint` sin `eyebrow/value/label`, queda desalineado. Edge case menor.
10. **`ProgressSummary.jsx:30-31`** — renderiza `footer` en dos elementos separados (divider + div), duplicando la condición `{footer ? … : null}`. Un solo fragmento lo expresaría con un render.
