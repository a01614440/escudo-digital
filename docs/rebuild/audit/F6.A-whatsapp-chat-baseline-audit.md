## Auditoria breve del estado

F6.A si hace falta porque F5 cerro el marco, pero no la simulacion especifica. El contenedor inmersivo ya deja que `sim_chat` respire, pero el interior de `WhatsAppSimulation` todavia tiene deuda propia: mezcla nueva/legacy, texto con posible mojibake, soporte/feedback no del todo integrado al hilo y un bloque legacy inaccesible que sigue dentro del archivo.

No conviene saltar directo a SMS, Inbox o WebLab porque `WhatsApp / chat` fija el patron base `Thread + Message + Composer + Inspector`. Si este canal no queda solido, SMS y ScenarioFlow heredaran decisiones flojas.

Tampoco conviene tocar `ActivityRenderer` ni `activityRegistry`: ambos ya estan marcados como contratos congelados. F6.A debe trabajar alrededor de `sim_chat` dentro de su componente y estilos, preservando el contrato existente.

## 1. Estado actual de WhatsAppSimulation

### Que esta bien

- `sim_chat` ya resuelve por contrato a `WhatsAppSimulation` desde `activityRegistry.js`.
- `ActivityRenderer.jsx` ya pasa `viewport`, `module`, `activity`, `answers`, `assessment`, `startedAtRef` y `onComplete` sin acoplarse a detalles del canal.
- `sharedActivityUi.jsx` ya trata `sim_chat` como actividad inmersiva y omite el chrome generico, dejando `sd-chat-activity-shell`.
- La implementacion activa de `WhatsAppSimulation` ya usa primitives nuevas: `SurfaceCard`, `Button`, `Input`, `InlineMessage`, `Badge`, `ActionCluster`, `PanelHeader`.
- Hay estructura base de chat: header de contacto, hilo, bubbles, typing, quick replies, composer, soporte y feedback.
- El servicio funcional ya esta aislado en `requestSimulationTurn`; F6.A no necesita tocar contratos API para mejorar la experiencia visual.
- `tailwind.css` ya contiene un bloque nuevo `sd-chat-*` para la version activa.

### Que esta mal

- `WhatsAppSimulation` sigue siendo hibrido. La version activa usa `sd-chat-*`, pero despues del primer `return` existe otro `return` legacy con `wa-*`, `FeedbackPanel`, botones `.btn` y `ActivitySummaryBar`. Ese bloque es inaccesible, pero mantiene deuda, imports y confusion para la siguiente edicion.
- El componente mezcla muchas responsabilidades en un solo bloque: derivacion de senales, estado de conversacion, request async, feedback, UI del hilo, UI del inspector, completion y responsive.
- Hay strings visibles con posible mojibake en literales estaticos. Algunos pasan por `repairPossibleMojibake`, pero no todos.
- `legacyThreatNote` queda calculado pero no se usa en la implementacion activa.
- `ActivitySummaryBar` y `FeedbackPanel` se importan por el bloque legacy inaccesible, no por la UI activa.
- El hilo tiene `aria-live="polite"`, pero no hay una region semantica clara que separe historial, mensaje nuevo, estado typing y composer.
- El composer no esta garantizado como fijo o sticky en mobile. Visualmente existe al final del hilo, pero no queda protegido como accion persistente de mensajeria movil.
- Quick replies compiten con composer y no tienen una jerarquia clara entre "respuesta sugerida" y "mensaje propio".
- Feedback y senales todavia se sienten como tarjetas educativas alrededor del hilo; F6.A debe decidir como integrarlas sin volver al dashboard.
- Desktop permite inspector, pero la anchura general de `.sd-chat-sim` esta limitada a `46rem`, lo cual reduce el valor del inspector persistente y deja la experiencia menos fullscreen de lo que F5 preparo.
- `app.css` y `legacy.css` conservan estilos `wa-*` que pertenecen a la version vieja. No deben limpiarse masivamente en F6.A, pero si hay que evitar reactivar esa ruta.

### Deuda viva

- Falta separar conceptualmente `Thread`, `Message`, `Composer` e `Inspector`, aunque sea dentro del mismo archivo si no se crean nuevos componentes todavia.
- Falta una politica responsive especifica:
  - mobile: hilo protagonista + composer fijo/sticky abajo + soporte secundario;
  - tablet: hilo dominante + inspector contextual no invasivo;
  - desktop: hilo amplio + inspector persistente sin aplastar el contenido.
- Falta guardrail de test especifico F6.A que proteja que no regrese el segundo `return` legacy, que `ActivityRenderer`/`activityRegistry` no se toquen y que existan hooks `data-sd-*` del chat.
- Falta closeout de a11y especifico del chat: live region, labels de hilo, estado typing, boton de cerrar, composer y quick replies.

## 6. Riesgos principales

### Visuales

- Que el chat siga pareciendo una tarjeta educativa y no mensajeria.
- Que el inspector desktop robe ancho al hilo.
- Que el composer mobile no quede lo bastante presente.
- Que los restos `wa-*` contaminen decisiones nuevas.

### Funcionales

- Romper el flujo async `requestSimulationTurn` por refactor de UI.
- Cambiar sin querer el shape de `history`, `turn`, `turnos_max` o completion details.
- Romper completion si se separan componentes sin pasar `startedAtRef` / `onComplete` correctamente.

### A11y

- Anunciar demasiados cambios en live region.
- No etiquetar adecuadamente quick replies, typing y composer.
- Dejar feedback importante sin relacion con el hilo o el estado de cierre.

### Scope creep

- Tocar SMS/Inbox porque comparten idea conversacional.
- Reabrir Lesson shell por incomodidades que deben resolverse dentro de `sim_chat`.
- Usar `ActivityRenderer` como atajo de layout.
- Limpiar `app.css`/`legacy.css` antes de F8.

## 7. Recomendacion final

La estructura recomendada para F6.A es correcta si se mantiene estrictamente como WhatsApp / chat:

- F6.A.1 Thread + message layout cleanup
- F6.A.2 Composer + quick replies pass
- F6.A.3 Inspector + signals + feedback pass
- F6.A.4 Responsive + a11y hardening
- F6.A.5 F6.A closeout validation

No recomiendo abrir SMS, Inbox, ScenarioFlow, CallSimulation ni WebLab hasta cerrar F6.A. WhatsApp debe convertirse en la referencia de `Thread + Message + Composer + Inspector`; luego las otras simulaciones pueden derivar o contrastar contra ese patron sin copiarlo ciegamente.
